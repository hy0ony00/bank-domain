2. Git으로 clone

이게 더 제대로 된 방법이야.

CMD에서:

git clone 저장소주소

예:

git clone https://github.com/아이디/banking-domain.git

그다음:

cd banking-domain

이후에 아까 준 mkdir, type nul > ... 명령어 실행하면 돼.

수정 끝나면:

git add .

git push

그러면 웹 GitHub 저장소에도 그대로 반영돼.

처음이면 ZIP보다 clone 방식이 훨씬 편해. 수정하고 바로 push하면 되거든.

clone은 다운안받은상태에서 가능 ?

응. 아직 로컬에 안 받은 상태에서 git clone 하는 거야.

clone 자체가 GitHub 저장소를 내 컴퓨터로 처음 내려받는 명령어야.

예를 들어 CMD에서:

cd C:\Users\사용자명\Documents
git clone https://github.com/깃허브아이디/banking-domain.git

banking-domain

그다음:

cd banking-domain

'git'은(는) 내부 또는 외부 명령, 실행할 수 있는 프로그램, 또는

winget install --id Git.Git -e --source winget

git --version

자기자본이 머야

자기자본은 쉽게 말하면 은행이 자기 돈으로 가지고 있는 자금이야.

은행 돈은 크게 두 종류로 볼 수 있어.

고객이 맡긴 예금: 나중에 고객에게 돌려줘야 하는 돈
자기자본: 은행 주주가 넣은 돈과 은행이 벌어서 쌓아둔 돈

예를 들어 은행에 돈이 총 10조 원 있어도, 그중 9조 원이 고객 예금이라면 그 돈 전부를 은행 자기 돈이라고 볼 수는 없어.

고객 예금 등 부채: 9조 원
은행 자기자본: 1조 원
총자산: 10조 원

대략 이런 구조야.

왜 자기자본이 중요한가

대출해준 돈을 못 받으면 은행에 손실이 생기잖아.

그 손실을 우선 자기자본으로 버텨야 해.

대출 부실 발생
→ 은행 손실 발생
→ 자기자본에서 손실 흡수

그래서 자기자본이 많을수록 은행이 손실을 견딜 힘이 크다고 볼 수 있어.

네가 본 여신 한도에서의 의미

예를 들어 은행 자기자본이 1조 원이고, 동일인 한도가 자기자본의 20%라면:

1조 원 × 20% = 2,000억 원

특정 개인이나 법인에게 2,000억 원을 초과해서 신용공여하지 못한다는 의미야.

자기자본을 기준으로 한도를 정하는 이유는 은행의 체력에 맞게 대출 위험을 제한하기 위해서야.

개인으로 비유하면

개인이 집을 샀다고 가정하면:

집값: 5억 원
주택담보대출: 4억 원
내 돈: 1억 원

여기서 개인의 자기자본과 비슷한 개념은 내 돈 1억 원이야.

집값이 떨어져도 내 돈이 어느 정도 완충 역할을 하는 것처럼, 은행도 자기자본으로 손실을 버텨.

자기자본은 은행이 손실을 감당할 수 있는 기본 체력이다.

1. 템플릿 문법

Vue 화면은 보통 HTML처럼 생긴 template과 JavaScript 로직으로 나뉘어 있어.

<div>
    <p>{{ message }}</p>
</div>
data() {
    return {
        message: "안녕하세요"
    }
}

{{ message }}는 data() 안의 값을 화면에 출력하는 문법이야.

화면에는 이렇게 나와.

안녕하세요

템플릿에서 자주 보는 건 이 정도야.

{{ value }}
v-bind
v-on
v-if
v-show
v-for
v-model

Vue 템플릿은 HTML과 컴포넌트 데이터를 연결하는 역할을 해.

2. v-bind

v-bind는 Vue 데이터 값을 HTML 속성이나 컴포넌트 속성에 연결하는 문법이야.

<button v-bind:disabled="isDisabled">
    저장
</button>
data() {
    return {
        isDisabled: true
    }
}

isDisabled가 true이면 버튼이 비활성화돼.

보통 축약해서 :만 써.

<button :disabled="isDisabled">
    저장
</button>

둘은 완전히 같은 의미야.

v-bind:disabled="isDisabled"
:disabled="isDisabled"
컴포넌트에서 자주 쓰는 모습
<c-grid
    :option="gridOption"
    :data="gridData"
/>

뜻은:

부모 컴포넌트의 gridOption을
c-grid의 option이라는 값으로 전달

부모 컴포넌트의 gridData를
c-grid의 data라는 값으로 전달

즉, :를 보면 일단 이렇게 생각하면 돼.

오른쪽에 있는 JavaScript 값을 왼쪽 속성에 연결한다.

문자열과 변수 차이
<c-input value="userName" />

이건 문자열 "userName"을 전달해.

<c-input :value="userName" />

이건 data() 안의 userName 값을 전달해.

이 차이 엄청 중요해.

3. v-on

v-on은 클릭, 입력, 변경 같은 이벤트가 발생했을 때 메서드를 실행하는 문법이야.

<button v-on:click="save">
    저장
</button>

보통 축약해서 @를 써.

<button @click="save">
    저장
</button>

JavaScript는:

methods: {
    save() {
        console.log("저장");
    }
}

흐름은:

사용자가 버튼 클릭
→ click 이벤트 발생
→ save() 실행
자주 보는 이벤트
@click="search"
@change="changeValue"
@input="inputValue"
@keyup.enter="search"
@focus="focusInput"
@blur="blurInput"

예를 들어:

<c-button @click="selectList">
    조회
</c-button>

오류 분석할 때는 이렇게 따라가면 돼.

화면에서 문제가 난 버튼을 찾음
@click에 연결된 메서드명을 봄
methods에서 해당 메서드를 찾음
그 안에서 호출하는 함수나 API를 따라감
<button @click="search">
methods: {
    search() {
        this.selectList();
    },

    selectList() {
        // API 호출
    }
}
4. v-if

v-if는 조건에 따라 HTML 요소나 컴포넌트를 실제로 만들거나 제거하는 문법이야.

<p v-if="isVisible">
    화면에 표시됩니다.
</p>
data() {
    return {
        isVisible: true
    }
}

true이면 나타나고 false이면 DOM 자체가 없어져.

<p v-if="status === 'APRV'">
    승인 완료
</p>

<p v-else-if="status === 'RJCT'">
    반려
</p>

<p v-else>
    결재 진행 중
</p>

v-if가 꺼지면 해당 요소나 하위 컴포넌트는 제거되고, 다시 켜지면 새로 생성돼.

v-if와 v-show 차이
<div v-if="visible">내용</div>

false면 DOM에서 없어져.

<div v-show="visible">내용</div>

false여도 DOM에는 있고 CSS의 display: none으로 숨겨져.

쉽게:

v-if   = 아예 만들거나 없앰
v-show = 만들어둔 상태에서 숨김

오류 분석할 때 화면에 컴포넌트가 안 보이면 먼저 봐야 할 게:

v-if
v-show

그리고 그 조건값이 실제로 뭐인지 확인해야 해.

5. Props

props는 부모 컴포넌트가 자식 컴포넌트에 값을 전달하는 방법이야.

부모:

<c-grid
    :data="gridData"
    :option="gridOption"
/>

자식 c-grid:

props: {
    data: {
        type: Array
    },
    option: {
        type: Object,
        required: true
    }
}

흐름은:

부모의 gridData
→ 자식의 data prop

부모의 gridOption
→ 자식의 option prop

Vue 컴포넌트는 자신이 받을 props를 명시적으로 선언할 수 있고, 타입이나 필수 여부도 지정할 수 있어.

네가 봤던 코드로 보면
props: {
    id: {
        type: String,
        required: true
    },
    option: {
        type: Object,
        required: true
    },
    data: {
        type: Array
    }
}

뜻은:

id
- 문자열이어야 함
- 반드시 넘겨야 함

option
- 객체여야 함
- 반드시 넘겨야 함

data
- 배열이어야 함
- 필수는 아님

호출하는 쪽은:

<c-grid
    id="grdMain"
    :option="gridOption"
    :data="gridData"
/>
Props 오류를 볼 때

컴포넌트 값이 이상하면 아래를 순서대로 봐.

1. 부모에서 값을 제대로 넘겼나
2. 속성명이 자식 props 이름과 같은가
3. :를 붙였나
4. 타입이 맞나
5. 값이 아직 undefined인 시점은 아닌가

예를 들어:

<c-grid data="gridData" />

이건 배열이 아니라 문자열을 보낸 거야.

올바른 건:

<c-grid :data="gridData" />
6. 커스텀 이벤트

Props가 부모에서 자식으로 값을 내리는 방식이라면, 커스텀 이벤트는 자식이 부모에게 어떤 일이 발생했다고 알리는 방식이야.

방향을 기억하면 돼.

Props
부모 → 자식

커스텀 이벤트
자식 → 부모
자식 컴포넌트
methods: {
    selectRow(row) {
        this.$emit("select", row);
    }
}

$emit()은 이벤트를 발생시키는 거야.

select라는 이벤트 발생
row 데이터를 같이 전달
부모 컴포넌트
<user-popup @select="setUser" />
methods: {
    setUser(row) {
        this.userId = row.userId;
        this.userName = row.userName;
    }
}

전체 흐름:

자식 팝업에서 행 선택
→ this.$emit("select", row)
→ 부모의 @select가 받음
→ setUser(row) 실행

Vue 컴포넌트 간 통신에서 props는 하향 데이터 전달, 컴포넌트 이벤트는 자식에서 부모로 알림을 전달하는 핵심 방식이야.

실무에서 자주 보는 형태
this.$emit("close");
this.$emit("change", value);
this.$emit("input", value);
this.$emit("select", row);
this.$emit("save", data);

부모에서는:

<custom-popup
    @close="closePopup"
    @select="setSelectedData"
/>

오류 분석할 때 팝업에서 값을 골랐는데 부모 화면에 반영이 안 되면:

1. 자식에서 $emit을 했는지
2. 이벤트 이름이 같은지
3. 부모가 @이벤트명을 받고 있는지
4. 넘긴 파라미터가 있는지
5. 부모 메서드가 값을 제대로 세팅하는지

이 순서로 보면 돼.

7. 생명주기

생명주기는 Vue 컴포넌트가 만들어지고, 화면에 붙고, 변경되고, 사라지는 과정이야.

Vue 컴포넌트는 생성될 때 데이터 설정, 템플릿 컴파일, DOM 마운트, 업데이트 등의 단계를 거치며, 각 단계에 맞는 생명주기 훅을 제공해.

Vue 2에서 주로 보는 건:

beforeCreate
created
beforeMount
mounted
beforeUpdate
updated
beforeDestroy
destroyed

실무에서는 우선 아래 네 개 위주로 보면 돼.

created

컴포넌트 객체와 데이터는 만들어졌지만, 화면 DOM은 아직 붙기 전이야.

created() {
    this.selectCodeList();
}

주로:

초기 데이터 세팅
공통코드 조회
파라미터 확인
API 조회 준비

같은 걸 해.

다만 DOM이나 $refs를 직접 사용하기에는 이른 시점이야.

created() {
    console.log(this.$refs.grid); // 보통 아직 없음
}
mounted

컴포넌트가 실제 DOM에 붙은 뒤야.

mounted() {
    this.createGrid();
}

주로:

그리드 생성
DOM 크기 계산
포커스 지정
$refs 접근
외부 UI 라이브러리 초기화

에 사용해.

mounted() {
    this.$refs.input.focus();
}

네 프로젝트의 SlickGrid 같은 건 보통 mounted 이후에 만드는 이유가, 실제 그리드를 붙일 DOM이 필요하기 때문이야.

updated

데이터가 바뀌어 화면이 다시 그려진 뒤 실행돼.

updated() {
    console.log("화면 갱신 완료");
}

여기서 다시 데이터를 바꾸면 무한 반복이 생길 수 있어서 조심해야 해.

updated() {
    this.count++;
}

이런 건:

count 변경
→ 화면 업데이트
→ updated 실행
→ count 변경
→ 다시 업데이트

무한 반복 가능성이 있어.

beforeDestroy / destroyed

컴포넌트가 사라지기 전과 후에 실행돼.

beforeDestroy() {
    window.removeEventListener("resize", this.resize);
}

주로:

이벤트 제거
타이머 해제
외부 객체 정리
메모리 누수 방지

에 사용해.

생명주기 흐름
컴포넌트 생성
→ created
→ template 처리
→ DOM 연결
→ mounted
→ 데이터 변경
→ updated
→ 컴포넌트 제거
→ beforeDestroy
→ destroyed
8. 컴포넌트 오류 분석할 때 보는 순서

실제 파일을 열었을 때는 처음부터 모든 코드를 읽지 말고 이렇게 보면 돼.

화면에 값이 안 나오는 경우
1. template에서 어떤 변수명을 쓰는지 확인
2. data 또는 props에 그 값이 있는지 확인
3. created나 mounted에서 값이 세팅되는지 확인
4. API 결과가 실제로 들어오는지 확인
5. v-if 또는 v-show로 숨겨진 건 아닌지 확인
버튼을 눌러도 반응이 없는 경우
1. @click 메서드명 확인
2. methods에서 해당 메서드 찾기
3. 메서드 안에 조건문이나 return이 있는지 확인
4. disabled 상태인지 확인
5. v-if로 버튼이 다시 생성되는 구조인지 확인
부모에서 넘긴 값이 자식에 안 들어가는 경우
1. 부모의 :속성명 확인
2. 자식 props 이름 확인
3. :가 빠졌는지 확인
4. 타입 확인
5. 전달 시점에 값이 undefined인지 확인
자식에서 선택했는데 부모에 반영 안 되는 경우
1. 자식의 this.$emit 확인
2. 이벤트명 확인
3. 부모의 @이벤트 확인
4. 부모 methods 확인
5. 전달되는 파라미터 확인
그리드나 DOM 오류가 나는 경우
1. createGrid가 언제 호출되는지 확인
2. created인지 mounted인지 확인
3. 해당 DOM이 v-if로 아직 안 만들어진 상태인지 확인
4. this.$refs가 존재하는지 확인
5. 화면이 숨겨진 상태에서 크기를 계산한 건 아닌지 확인
딱 점검해야 할 수준

아래 코드를 보고 흐름을 설명할 수 있으면 일단 충분해.

<user-popup
    v-if="popupVisible"
    :user-id="userId"
    @select="setUser"
    @close="popupVisible = false"
/>
data() {
    return {
        popupVisible: false,
        userId: ""
    };
},

mounted() {
    this.selectUser();
},

methods: {
    selectUser() {
        this.userId = "E00001";
    },

    setUser(user) {
        this.userId = user.userId;
        this.popupVisible = false;
    }
}

해석하면:

1. 컴포넌트가 화면에 붙으면 mounted 실행
2. selectUser()가 실행돼 userId를 E00001로 설정
3. popupVisible이 true일 때만 user-popup 생성
4. userId를 자식의 user-id prop으로 전달
5. 자식이 select 이벤트를 발생시키면 setUser 실행
6. 선택된 userId를 세팅하고 팝업을 닫음

결국 핵심은 이거야.

:는 값을 넘기고, @는 이벤트를 받고, v-if는 컴포넌트를 만들거나 없애고, props는 부모 값을 받으며, $emit은 자식이 부모에게 알리고, 생명주기는 이 로직들이 언제 실행되는지를 결정한다.

v-bind
v-on
v-if
v-show
v-for
v-model 이거도 템플릿 문법이야 ?

응, 전부 Vue 템플릿 문법이야.

정확히는 HTML 안에서 Vue가 해석하는 디렉티브(directive)들이야.

<div v-if="visible">
    {{ message }}
</div>

여기서:

{{ message }} → 보간법, 텍스트 출력
v-if → 조건부 렌더링
v-bind → 속성에 데이터 연결
v-on → 이벤트 연결
v-show → 보이기/숨기기
v-for → 반복 출력
v-model → 입력값 양방향 연결

그래서 크게 보면:

Vue 템플릿 문법
├── 보간법 {{ }}
├── 디렉티브
│   ├── v-bind
│   ├── v-on
│   ├── v-if
│   ├── v-show
│   ├── v-for
│   └── v-model
└── 속성/이벤트 축약
    ├── :   = v-bind
    └── @   = v-on

예를 들면:

<input
    v-if="isVisible"
    v-model="userName"
    :disabled="isDisabled"
    @keyup.enter="search"
/>

해석하면:

isVisible이 true일 때만 input 생성
userName과 입력값 연결
isDisabled 값에 따라 비활성화
엔터를 누르면 search 실행

다만 props, methods, data, computed, 생명주기는 템플릿 문법은 아니고 컴포넌트 옵션/로직 영역이야.

인스턴스 같은건가 ?

아니, v-bind, v-on, v-if 같은 건 인스턴스가 아니라 디렉티브야.

구분하면 이렇게 보면 돼.

Vue 인스턴스 / 컴포넌트
└── template 안에서 Vue 문법 사용
    ├── v-bind
    ├── v-on
    ├── v-if
    ├── v-for
    └── v-model

예를 들어:

new Vue({
    el: "#app",
    data: {
        visible: true,
        message: "안녕하세요"
    }
});

이 new Vue({...})가 Vue 인스턴스야.

그리고 그 인스턴스가 관리하는 HTML 안에서:

<div id="app">
    <p v-if="visible">{{ message }}</p>
</div>

여기서:

v-if → 디렉티브
{{ message }} → 템플릿 문법
visible, message → Vue 인스턴스의 데이터

쉽게 말하면:

인스턴스는 화면과 데이터를 관리하는 주체고,
디렉티브는 그 화면을 어떻게 동작시킬지 적는 문법이야.

Vue 컴포넌트도 결국 실행되면 각각 하나의 Vue 인스턴스처럼 동작한다고 이해하면 돼.

props 

props는 부모 컴포넌트가 자식 컴포넌트에게 값을 넘겨줄 때 쓰는 것이야.

인스턴스도 아니고, 템플릿 디렉티브도 아니고, 컴포넌트 옵션에 가까워.

예를 들어 부모가 이렇게 넘겨:

<user-card :user-name="name" />

부모 쪽 데이터:

data() {
    return {
        name: "재윤"
    };
}

자식 컴포넌트에서는 이렇게 받아:

props: {
    userName: {
        type: String,
        required: true
    }
}

자식 템플릿에서 사용:

<p>{{ userName }}</p>

흐름은:

부모의 name
→ :user-name으로 전달
→ 자식의 props.userName으로 받음
→ 자식 화면에서 사용

핵심은 방향이야.

props
부모 → 자식

그리고 보통 자식은 props 값을 직접 바꾸지 않아.

이건 비추천:

this.userName = "홍길동";

왜냐하면 원래 부모가 넘긴 값이기 때문이야.

자식에서 바꾸고 싶으면 보통 이벤트를 올려:

this.$emit("change-name", "홍길동");

부모가 받아서 자기 데이터를 바꿔.

<user-card
    :user-name="name"
    @change-name="name = $event"
/>

한 줄로 정리하면:

props는 부모가 가진 데이터를 자식 컴포넌트에 전달하기 위한 입력값이다.

아 인터페이스 같은건가 props

비슷하게 느껴질 수는 있는데, Java의 interface랑 완전히 같은 건 아니야.

props는 더 정확히 말하면 컴포넌트가 외부에서 받을 수 있는 입력값 명세야.

예를 들어 자식 컴포넌트가:

props: {
    userName: {
        type: String,
        required: true
    },
    age: {
        type: Number,
        default: 0
    }
}

라고 선언하면, 이 컴포넌트는 외부에 이렇게 말하는 셈이야.

나는 userName이라는 문자열을 반드시 받아야 하고,
age라는 숫자는 선택적으로 받을 수 있어.

부모는 이렇게 넘기고:

<user-card
    :user-name="name"
    :age="userAge"
/>

그래서 개념적으로는 인터페이스보다는 메서드 파라미터에 더 가까워.

void printUser(String userName, int age)

이 메서드가 userName, age를 입력으로 받는 것처럼, Vue 컴포넌트도 props를 입력으로 받는 거야.

다만 props가 컴포넌트의 외부 사용 규칙을 정한다는 점에서는 컴포넌트 인터페이스 역할을 한다고 표현할 수 있어.

Java interface
→ 클래스가 구현해야 할 메서드 규칙

Vue props
→ 컴포넌트가 외부에서 받을 데이터 규칙

그래서 한 줄로 하면:

props는 Java 인터페이스 그 자체는 아니지만, 컴포넌트 입장에서는 외부에 공개된 입력 규격, 즉 컴포넌트 API 역할을 한다.

예를 들어 네 c-grid에서:

props: {
    id: {
        type: String,
        required: true
    },
    option: {
        type: Object,
        required: true
    },
    data: {
        type: Array
    }
}

이건 c-grid를 쓰려면:

id는 문자열로 필수
option은 객체로 필수
data는 배열로 선택

라는 사용 규칙이 되는 거야.

## 1. 여신의 의미

### 1-1. 법률적인 의미

여신은 은행이 고객에게 돈을 직접 빌려주는 **대출보다 넓은 개념**이다.

「은행법」에서는 여신, 즉 신용공여를 은행이 금융거래 과정에서 직접 또는 간접적으로 신용위험을 부담하는 거래라고 본다.

여신에는 다음과 같은 거래가 포함된다.

- 대출
- 어음할인
- 지급보증
- 유가증권 매입
- 유가증권 대여
- 그 밖에 은행이 신용위험을 부담하는 거래

여신인지 판단할 때 중요한 것은 은행의 돈이 지금 바로 지급되는지 여부가 아니다.

> 고객이나 거래상대방이 약속을 지키지 못했을 때 은행에 손실이 발생할 가능성이 있는지가 중요하다.

따라서 다음과 같이 정리할 수 있다.

> 모든 대출은 여신에 포함되지만, 모든 여신이 대출인 것은 아니다.

### 1-2. 여신거래의 예시

### 대출

은행이 고객에게 돈을 직접 빌려주고, 고객은 정해진 기간 동안 원금과 이자를 갚는 거래이다.

예를 들어 은행이 고객에게 1억 원을 빌려줬는데 고객이 원금과 이자를 갚지 못하면 은행에 손실이 발생한다.

### 지급보증

고객이 다른 사람이나 회사에 부담하는 의무를 지키지 못할 경우, 은행이 대신 지급하겠다고 약속하는 거래이다.

지급보증을 해준 시점에는 은행의 돈이 바로 나가지 않는다.

하지만 고객이 의무를 지키지 못하면 은행이 대신 돈을 지급해야 하므로 여신에 포함된다.

### 어음할인

어음은 일정한 날짜에 돈을 지급하겠다는 내용이 적힌 증서이다.

고객이 몇 개월 뒤 받을 어음을 가지고 있지만 지금 당장 돈이 필요한 경우가 있다.

이때 은행이 어음을 넘겨받고, 만기까지의 할인료를 뺀 금액을 고객에게 먼저 지급하는 거래가 어음할인이다.

은행은 어음 만기일에 어음에 적힌 금액을 받는다.

어음이 부도나거나 지급되지 않으면 은행이 돈을 회수하지 못할 수 있으므로 어음할인도 여신에 포함된다.

### 1-3. 은행이 지급보증을 하는 이유

은행은 지급보증을 제공하고 고객으로부터 보증료를 받는다.

고객이 약속한 의무를 정상적으로 이행하면 은행은 실제 자금을 지급하지 않고 보증료 수익을 얻을 수 있다.

지급보증의 흐름은 다음과 같다.

1. 고객이 계약 이행이나 채무 지급을 위해 은행에 지급보증을 신청한다.
2. 은행이 고객의 신용상태와 상환능력을 심사한다.
3. 심사 결과 문제가 없으면 은행이 보증서를 발급한다.
4. 고객은 은행에 보증료를 납부한다.
5. 고객이 의무를 정상적으로 이행하면 은행의 실제 자금지출은 발생하지 않는다.
6. 고객이 의무를 이행하지 못하면 은행이 대신 지급한다.
7. 은행은 대신 지급한 금액을 고객에게 다시 청구한다.

은행이 대신 지급한 금액을 고객이 갚지 못하면 은행에 손실이 발생한다.

따라서 지급보증은 다음 두 가지 성격을 함께 가진다.

- 은행이 보증료를 받을 수 있는 수익거래
- 고객이 의무를 이행하지 못하면 손실이 발생할 수 있는 여신거래

이 때문에 은행은 지급보증도 대출처럼 신용심사와 한도관리를 진행한다.

### 1-4. 어음할인과 대출의 차이

대출과 어음할인은 모두 은행이 고객에게 먼저 돈을 지급한다는 점에서는 비슷하다.

하지만 돈을 지급하는 근거와 회수방법이 다르다.

| 구분 | 대출 | 어음할인 |
|---|---|---|
| 거래의 근거 | 은행과 고객이 대출계약을 체결 | 고객이 보유한 어음을 은행이 매입 |
| 고객이 받는 돈 | 대출금 | 어음금액에서 할인료를 뺀 금액 |
| 은행의 수익 | 대출이자 | 할인료 |
| 원래 돈을 지급할 사람 | 대출받은 고객 | 어음에 적힌 지급인 |
| 주요 위험 | 고객이 원금과 이자를 갚지 못함 | 어음이 부도나거나 지급되지 않음 |

### 예시

A회사가 B회사에 물건을 판매하고, 3개월 뒤 1억 원을 지급받기로 한 어음을 받았다고 가정한다.

하지만 A회사는 지금 당장 운영자금이 필요하다.

A회사는 어음을 은행에 넘기고, 은행은 할인료를 제외한 9,800만 원을 먼저 지급한다.

3개월 뒤에는 B회사가 은행에 어음금액 1억 원을 지급한다.

이 거래가 어음할인이다.

반면 대출은 고객이 가진 어음을 은행에 넘기는 것이 아니라, 고객과 은행이 직접 대출계약을 체결하고 고객이 은행에 원금과 이자를 갚는 거래이다.

### 1-5. 여신의 기능

은행은 고객의 예금 등을 통해 자금을 마련하고, 그 자금을 대출과 같은 여신으로 운용한다.

기본적인 흐름은 다음과 같다.

> 고객에게서 예금을 받음  
> → 필요한 고객이나 기업에 자금을 빌려줌  
> → 원금과 이자를 돌려받음  
> → 은행의 수익이 발생함

### 은행 측면

은행 입장에서 여신은 자금을 운용하고 수익을 얻는 주요 업무이다.

은행은 예금 등을 통해 조달한 자금을 대출 등으로 운용하고, 이자와 수수료를 통해 수익을 얻는다.

### 국가경제 측면

국가경제 측면에서 여신은 돈이 남는 사람의 자금을 돈이 필요한 개인이나 기업에 전달하는 역할을 한다.

예를 들면 다음과 같다.

- 개인이 은행에 예금한다.
- 기업이 은행에서 대출받아 공장이나 장비를 구입한다.
- 개인이 주택구입자금을 대출받는다.
- 사업자가 운영자금을 대출받아 사업을 계속한다.

은행은 이러한 과정을 통해 다음과 같은 기능을 수행한다.

- 자금의 매개
- 신용의 창조
- 자금의 배분
- 경제활동 지원

### 1-6. 여신을 신중하게 취급해야 하는 이유

은행이 빌려주는 돈은 대부분 은행의 자기 돈이 아니라 고객이 맡긴 예금에서 나온다.

은행이 상환능력이 없는 고객에게 무리하게 대출하고 돈을 돌려받지 못하면 다음과 같은 문제가 발생할 수 있다.

- 은행에 손실이 발생한다.
- 은행의 재무상태가 나빠진다.
- 예금자에게 영향을 줄 수 있다.
- 은행에 대한 신뢰가 떨어질 수 있다.
- 금융시장 전체에 문제가 생길 수 있다.

따라서 은행은 여신을 취급할 때 다음 사항을 확인해야 한다.

- 고객이 돈을 갚을 능력이 있는가
- 고객이 돈을 갚으려는 의지가 있는가
- 계약이 법적으로 문제가 없는가
- 고객의 위험에 맞는 금리를 적용했는가
- 원금과 이자를 제때 받을 수 있는 상환방식인가
- 담보나 보증이 필요한가

여신업무의 핵심은 단순히 대출을 많이 실행하는 것이 아니다.

> 돈을 빌려줘도 되는 고객을 잘 판단하고, 빌려준 돈을 제대로 회수하는 것이 핵심이다.

### 1-7. 여신운용의 기본원칙

여신운용의 4대 기본원칙은 다음과 같다.

- 안정성
- 수익성
- 성장성
- 공공성

### 안정성의 원칙

빌려준 돈을 안전하고 확실하게 회수할 수 있어야 한다는 원칙이다.

은행은 대출을 실행하기 전에 다음 사항을 확인한다.

- 고객의 소득이나 매출
- 기존에 보유한 대출
- 원금과 이자를 갚을 능력
- 돈을 갚으려는 의지
- 담보와 보증
- 특정 고객이나 업종에 대출이 몰려 있는지 여부

담보가 있다고 무조건 안전한 대출은 아니다.

담보는 고객이 대출금을 갚지 못했을 때 처분하여 돈을 회수하는 수단이다.

담보를 처분하는 데 시간이 오래 걸릴 수 있고, 담보가격이 떨어질 수도 있다.

따라서 은행은 담보보다 먼저 고객이 정상적인 소득이나 영업활동으로 대출금을 갚을 수 있는지를 확인해야 한다.

### 수익성의 원칙

은행이 부담하는 위험에 맞는 적절한 수익을 얻어야 한다는 원칙이다.

신용도가 좋은 고객과 연체 가능성이 높은 고객에게 같은 조건으로 대출해주면 은행이 부담하는 위험을 제대로 반영하지 못하게 된다.

위험이 높은 고객에게는 다음과 같은 조건이 적용될 수 있다.

- 대출금리가 높아질 수 있다.
- 승인금액이 줄어들 수 있다.
- 추가 담보나 보증을 요구할 수 있다.
- 대출이 거절될 수 있다.

은행은 고객의 신용위험, 대출기간, 담보 여부 등을 고려하여 금리와 한도를 결정한다.

### 성장성의 원칙

여신이 은행과 거래고객의 성장에 도움이 되어야 한다는 원칙이다.

기업은 대출받은 돈을 다음과 같이 사용할 수 있다.

- 공장이나 설비 구입
- 사업장 확대
- 원재료 구입
- 신규 사업 추진
- 운영자금 확보

기업이 성장하면 대출금을 갚을 능력도 좋아질 수 있고, 은행도 우량고객과의 거래를 확대할 수 있다.

다만 성장 가능성이 있다는 이유만으로 무조건 대출하는 것은 아니다.

성장 가능성과 실제 상환 가능성을 함께 검토해야 한다.

### 공공성의 원칙

은행의 여신은 은행의 수익뿐 아니라 국민경제에 미치는 영향도 고려해야 한다는 원칙이다.

은행이 특정 고객이나 업종에 돈을 지나치게 몰아주면 해당 고객이나 업종에 문제가 생겼을 때 은행에도 큰 손실이 발생할 수 있다.

또한 불법이나 투기를 위한 자금을 무분별하게 공급해서도 안 된다.

은행은 자금이 필요한 곳에 돈이 균형 있게 공급되고 경제발전에 도움이 되도록 여신을 운용해야 한다.

### 핵심 정리

- 여신은 대출보다 넓은 개념이다.
- 모든 대출은 여신이지만 모든 여신이 대출은 아니다.
- 은행이 손실을 볼 가능성이 있는 신용거래는 여신에 포함된다.
- 지급보증은 고객이 의무를 이행하지 못하면 은행이 대신 지급할 수 있으므로 여신이다.
- 어음할인은 미래에 받을 어음대금을 은행을 통해 미리 현금으로 바꾸는 거래이다.
- 은행은 예금으로 마련한 자금을 여신으로 운용하여 수익을 얻는다.
- 여신의 4대 기본원칙은 안정성, 수익성, 성장성, 공공성이다.
- 여신의 핵심은 상환 가능한 고객을 판단하고 빌려준 돈을 안전하게 회수하는 것이다.

## 2. 여신취급 금지 및 제한

---

### 여신취급을 제한하는 이유

은행은 고객이 원한다고 해서 무조건 돈을 빌려줄 수 없다.

은행이 빌려주는 돈의 대부분은 고객에게서 받은 예금이기 때문에, 위험한 고객이나 특정 기업에 대출이 지나치게 몰리면 은행의 건전성이 나빠질 수 있다.

따라서 다음과 같은 목적을 위해 법과 은행 내부규정으로 여신취급을 제한한다.

- 대출금이 특정 개인이나 기업에 지나치게 몰리는 것을 방지
- 은행의 대주주나 임직원이 특혜를 받는 것을 방지
- 상환 가능성이 매우 낮은 고객에게 추가 대출이 나가는 것을 방지
- 은행의 손실과 부실을 예방
- 예금자와 금융시장을 보호

여신취급 제한은 여신의 4대 기본원칙 중 특히 **안정성**과 **공공성**을 지키기 위한 제도이다.

### 2-1. 여신 관련 법규 및 규정

은행의 여신업무는 은행이 자체적으로 정한 기준만으로 처리하지 않는다.

은행은 외부 법규를 지켜야 하며, 각 은행은 해당 법규를 반영하여 내부 여신규정과 업무지침을 만든다.

여신업무와 관련된 대표적인 외부규정은 다음과 같다.

- 「은행법」
- 「은행법 시행령」
- 은행업감독규정
- 은행업감독업무시행세칙
- 그 밖의 금융 관련 법률 및 감독규정

이를 쉽게 구분하면 다음과 같다.

| 구분 | 의미 |
|---|---|
| 외규 | 법률, 시행령, 감독규정 등 은행 외부에서 정한 규정 |
| 내규 | 외규를 반영하여 각 은행이 내부적으로 만든 규정과 지침 |

외규에서 금지하는 거래는 은행 내부규정으로 허용할 수 없다.

은행 내부규정은 외부 법규를 기본으로 하면서, 은행의 위험관리정책에 따라 더 엄격하게 정할 수도 있다.

### 예시

법에서는 일정 한도까지 대출이 가능하더라도, 해당 은행의 내부정책상 위험이 높다고 판단되면 더 적은 금액만 승인하거나 대출을 거절할 수 있다.

---

### 2-2. 은행법 등 관련 법규에 따른 제한

은행은 한 고객이나 특정 관계자에게 지나치게 많은 대출을 해서는 안 된다.

특정 고객에게 대출이 몰린 상태에서 그 고객이 부실해지면 은행에 큰 손실이 발생할 수 있기 때문이다.

이를 막기 위해 동일인, 동일차주, 대주주 등에 대한 신용공여 한도를 정하고 있다.

---

### 동일인과 동일차주에 대한 한도

#### 동일인

동일인은 한 명의 개인 또는 하나의 법인을 의미한다.

예를 들어 다음은 각각 하나의 동일인이다.

- 개인 고객 한 명
- A주식회사
- B재단법인

#### 동일차주

동일차주는 법적으로는 서로 다른 개인이나 법인이지만, 실질적으로 같은 신용위험을 공유하는 관계자들을 하나로 묶은 개념이다.

예를 들어 다음과 같은 경우이다.

- 같은 기업집단에 속한 회사들
- 서로 지급보증을 제공하는 관계회사
- 한 회사가 부실해지면 다른 회사도 함께 영향을 받는 관계
- 실질적으로 동일한 경영주가 지배하는 여러 회사

### 왜 따로 묶어서 보는가?

A회사와 B회사가 서로 다른 법인이라는 이유만으로 각각 큰 대출을 해주면, 실제로는 같은 기업집단에 대출이 몰릴 수 있다.

따라서 은행은 단순히 법인등록번호만 따로 보는 것이 아니라, 실질적으로 위험을 함께 부담하는 관계인지 확인한다.

### 한도

- 동일한 개인 또는 법인 각각에 대한 신용공여: 은행 자기자본의 20% 초과 금지
- 동일차주에 대한 신용공여: 은행 자기자본의 25% 초과 금지

### 예시

은행의 자기자본이 1조 원이라고 가정한다.

- 동일인 한도 20%: 최대 2,000억 원
- 동일차주 한도 25%: 최대 2,500억 원

A회사에 이미 1,500억 원을 대출했고, A회사와 동일차주인 B회사에 1,000억 원을 추가로 대출하려 한다면 동일차주 전체 금액은 2,500억 원이 된다.

이보다 더 대출하면 동일차주 한도를 초과할 수 있다.

> 실제 프로젝트에서는 현재 법령과 해당 은행의 내부규정을 기준으로 확인해야 한다.

---

### 거액 신용공여 한도

은행 자기자본의 10%를 초과하는 신용공여를 **거액 신용공여**라고 한다.

거액 신용공여의 총합계가 은행 자기자본의 5배를 넘지 못하도록 규제한다고 한다.
이는 개별 고객에 대한 한도뿐 아니라, 큰 대출 전체가 은행 자본에 비해 지나치게 많아지는 것도 막기 위한 규정이다.

### 예시


은행의 자기자본이 1조 원이라면 다음과 같다.

- 10%인 1,000억 원을 초과하는 여신은 거액 신용공여
- 거액 신용공여 전체 합계는 5조 원을 초과할 수 없음

즉 고객별 한도를 지켰더라도 큰 대출이 너무 많이 쌓이면 은행 전체가 위험해질 수 있으므로 총량도 관리한다.

---

### 대주주에 대한 신용공여 제한

대주주는 은행의 주요 주식을 보유하여 은행 경영에 영향력을 행사할 수 있는 사람이나 법인이다.

은행이 대주주에게 마음대로 큰 금액을 빌려주게 되면 다음과 같은 문제가 발생할 수 있다.

- 대주주가 자신의 지위를 이용하여 특혜대출을 받을 수 있음
- 정상적인 심사 없이 대출이 승인될 수 있음
- 대주주가 부실해질 경우 은행도 함께 위험해질 수 있음
- 예금자의 돈이 대주주의 사적인 이익을 위해 사용될 수 있음

따라서 대주주 개인뿐 아니라 전체 대주주에게 제공할 수 있는 신용공여에도 한도를 둔다.

핵심은 다음과 같다.

> 은행의 주인이 은행에 영향력을 행사하여 자기에게 유리한 대출을 받지 못하도록 제한하는 제도이다.

---

### 은행법상 금지업무

#### 해당 은행의 주식을 담보로 하는 대출

고객이 그 은행의 주식을 담보로 제공하고 같은 은행에서 돈을 빌리는 거래이다.

은행의 주가가 떨어지면 담보가치와 은행 건전성이 동시에 나빠질 수 있다.

#### 해당 은행의 주식을 사기 위한 대출

은행이 고객에게 돈을 빌려주고, 고객이 그 돈으로 해당 은행의 주식을 사게 하는 거래이다.

이는 은행이 대출을 이용하여 자기 주가나 자기자본을 인위적으로 부풀릴 위험이 있다.

#### 해당 은행 임직원에 대한 대출

은행 임직원은 자신의 업무상 지위를 이용하여 대출심사나 조건에 영향을 줄 수 있다.

따라서 특혜대출을 막기 위해 임직원 대출을 제한한다.

다만 금융위원회가 정한 소액대출 등 일부 예외는 존재할 수 있다.

### 2-3. 신용도판단정보 및 공공정보 등록 대상자

은행은 여신을 취급하기 전에 고객에게 심각한 연체나 공공기관 체납 등의 문제가 있는지 확인해야 한다.

이를 위해 한국신용정보원 등을 통해 신용도판단정보와 공공정보를 조회한다.

해당 정보가 있다고 해서 모든 경우에 법적으로 무조건 대출이 금지되는 것은 아니다.

은행은 정보의 종류, 등록사유, 해제 여부, 내부규정 등을 확인하여 다음과 같이 처리할 수 있다.

- 신규대출 제한
- 기한연장 제한
- 심사 강화
- 대출한도 축소
- 추가 담보나 보증 요구
- 대출 거절

---

### 신용도판단정보

신용도판단정보는 금융거래 과정에서 고객이 돈을 제대로 갚지 않았거나 금융질서를 위반한 사실과 관련된 정보이다.

대표적인 정보는 다음과 같다.

- 연체정보
- 대위변제·대지급정보
- 부도정보
- 금융질서문란정보
- 관련인정보
- 미수발생정보
- 신용거래의 무담보 미수채권정보

#### 연체정보

대출 원금이나 이자, 카드대금 등을 약속한 날짜에 갚지 않은 정보이다.

#### 대위변제·대지급정보

보증기관이나 은행 등이 고객을 대신하여 돈을 지급한 정보이다.

예를 들어 보증기관이 고객 대신 은행에 대출금을 갚았다면 대위변제가 발생한다.

#### 부도정보

어음이나 수표가 지급되지 않은 정보이다.

#### 금융질서문란정보

대출사기, 명의도용, 허위서류 제출 등 금융질서를 심각하게 해친 정보이다.

#### 관련인정보

부도기업의 대표자처럼 부실 거래와 밀접하게 관련된 사람에 대한 정보이다.

### 쉽게 정리하면

> 신용도판단정보는 금융거래에서 돈을 갚지 않았거나 금융질서를 위반한 기록이다.

---

### 공공정보

공공정보는 금융회사뿐 아니라 국가기관이나 공공기관이 보유한 체납, 회생, 파산 등의 정보이다.

대표적인 정보는 다음과 같다.

- 국세 체납
- 지방세 체납
- 과태료 체납
- 관세 체납
- 산재보험료·고용보험료 체납
- 채무불이행자명부 등재
- 회생절차 진행
- 개인회생절차 진행
- 파산면책 결정
- 신용회복지원 확정

### 신용도판단정보와 공공정보의 차이

| 구분 | 신용도판단정보 | 공공정보 |
|---|---|---|
| 주된 발생영역 | 금융거래 | 국가기관·공공기관의 행정 및 사법절차 |
| 대표적인 예 | 연체, 부도, 대위변제 | 세금체납, 회생, 파산, 신용회복 |
| 판단 의미 | 금융거래를 정상적으로 이행했는지 | 공적 채무나 법적 채무조정 상태가 있는지 |

### 예시

고객이 은행 대출이자를 오래 갚지 않았다면 **연체정보**에 해당한다.

고객이 국세를 내지 않았다면 **공공정보**에 해당한다.

고객이 개인회생절차를 진행하고 있다면 이 또한 **공공정보**에 해당한다.

### 2-4. 신용등급 불량기업

신용등급이 매우 낮은 기업은 대출금을 갚지 못할 가능성이 높다.

따라서 은행은 신용등급이 불량한 기업에 대해 다음과 같이 처리할 수 있다.

- 신규여신 취급 제한
- 기존 대출 회수
- 대출한도 축소
- 추가대출 중단
- 신용등급이 개선될 때까지 여신 감축
- 추가 담보나 보증 요구

여기서 중요한 점은 신용등급이 낮다고 무조건 기업이 망했다는 뜻은 아니라는 것이다.

하지만 은행 입장에서는 부실 가능성이 높아졌다는 신호이므로 더 보수적으로 심사한다.

### 예시

A기업의 매출이 크게 줄고 부채가 늘어나 신용등급이 하락했다.

이 경우 은행은 기존 한도대출을 그대로 연장하지 않고 다음과 같은 조건을 붙일 수 있다.

- 한도 일부 감액
- 일부 원금 상환
- 추가 담보 제공
- 만기 단축
- 신규대출 중단

### 2-5. 연체대출금 보유자 또는 은행에 손실을 끼친 자

대출을 이미 연체하고 있거나 과거에 은행에 실제 손실을 끼친 고객에게 다시 돈을 빌려주면 은행의 손실이 더 커질 수 있다.

따라서 연체를 정리하거나 은행의 손실을 회복하지 않은 고객은 원칙적으로 신규여신이 제한된다.

은행은 여신취급 전에 다음 사항을 확인한다.

- 현재 연체대출금을 보유하고 있는지
- 원금이나 이자를 미납했는지
- 기한의 이익을 상실한 대출이 있는지
- 과거에 원금을 감면받은 사실이 있는지
- 특수채권으로 관리된 이력이 있는지
- 은행에 실제 손실을 발생시킨 이력이 있는지

---

### 연체대출금의 범위

연체대출금은 단순히 대출 만기일이 지났는데 갚지 않은 경우만을 의미하지 않는다.

다음과 같은 경우도 포함된다.

- 상환기일이 지났지만 회수되지 않은 대출금
- 이자를 납부하지 않아 연체처리된 대출금
- 분할상환일에 갚지 않은 분할상환금
- 지급보증 후 은행이 대신 지급한 대지급금
- 그 밖에 기한의 이익을 상실한 대출금

### 예시

매월 100만 원씩 갚는 분할상환대출에서 이번 달 상환금 100만 원을 납부하지 않았다면, 해당 분할상환금은 연체에 해당한다.

대출 만기는 아직 2년 남았더라도 약정 위반으로 기한의 이익을 상실했다면 전체 대출금이 연체대출금으로 처리될 수 있다.

---

### 기한의 이익 상실

#### 기한의 이익

대출 만기일까지는 채무자가 대출금 전액을 갚지 않아도 되는 이익이다.

예를 들어 대출 만기가 2028년 12월 31일이라면, 채무자가 약정을 정상적으로 지키는 동안 은행은 그 전에 대출금 전액을 갚으라고 요구할 수 없다.

#### 기한의 이익 상실

연체나 약정 위반 등 정해진 사유가 발생하여 만기 전이라도 채무자가 대출금 전액을 즉시 갚아야 하는 상태이다.

#### 발생 가능한 효과

- 대출금이 연체대출금으로 분류됨
- 채무자에게 대출금 전액을 즉시 청구할 수 있음
- 보증인에게 변제를 청구할 수 있음
- 채무자의 예금과 대출금을 상계할 수 있음
- 담보권을 실행할 수 있음
- 강제집행 등 채권회수 절차를 진행할 수 있음

### 쉽게 정리하면

> 원래는 만기까지 기다려주기로 했지만, 고객이 약속을 어겼기 때문에 은행이 지금 전액을 갚으라고 요구할 수 있게 되는 것이다.

### 예외적인 여신취급

연체대출금을 보유한 고객은 원칙적으로 신규여신이 제한된다.

다만 새로운 여신을 취급해야 기존 대출을 정상화하거나 회수할 수 있는 특별한 상황이라면 은행 내부기준에 따라 예외취급이 가능할 수 있다.

예를 들어 기업이 일시적인 자금부족을 겪고 있지만 추가 운전자금을 지원하면 정상 영업과 기존 대출 상환이 가능하다고 판단되는 경우이다.

단, 이러한 예외취급은 일반적인 승인이 아니라 별도의 심사와 승인절차가 필요하다.

[목차로 이동](#목차)

### 2-6. 부실회계기업

기업여신을 심사할 때 은행은 기업이 제출한 재무제표를 바탕으로 매출, 자산, 부채, 이익 등을 분석한다.

그런데 기업이 재무제표를 실제보다 좋게 보이도록 조작하면 은행은 잘못된 정보를 바탕으로 대출을 승인할 수 있다.

이러한 회계조작을 일반적으로 **분식회계**라고 한다.

### 예시

실제로는 손실이 발생했는데 매출이나 이익을 부풀려 흑자인 것처럼 표시하는 경우이다.

또는 실제 부채를 재무제표에서 누락하여 재무상태가 좋은 것처럼 보이게 할 수 있다.

이런 기업에 대출하면 은행은 실제 상환능력을 제대로 판단할 수 없으므로 위험하다.

### 감사의견

외부감사인은 기업의 재무제표를 확인한 뒤 감사의견을 제시한다.

대표적인 감사의견은 다음과 같다.

| 감사의견 | 의미 |
|---|---|
| 적정 | 재무제표가 중요한 부분에서 회계기준에 맞게 작성됨 |
| 한정 | 일부 문제는 있지만 재무제표 전체를 부정할 정도는 아님 |
| 부적정 | 재무제표의 문제가 매우 커서 적정하다고 볼 수 없음 |
| 의견거절 | 자료부족 등으로 감사인이 의견을 낼 수 없음 |

감사의견이 한정, 부적정 또는 의견거절이라면 은행은 해당 기업의 회계상 문제를 자세히 확인해야 한다.

특히 금융감독원으로부터 부실회계기업으로 지정된 기업은 여신취급이 제한될 수 있다.

### 왜 의견거절도 위험한가?

의견거절은 반드시 분식회계가 확인됐다는 뜻은 아니다.

하지만 감사인이 필요한 자료를 충분히 확인하지 못했거나 기업의 계속기업 가능성 등에 큰 불확실성이 있어 판단 자체를 할 수 없다는 의미일 수 있다.

은행 입장에서는 기업의 실제 재무상태를 믿기 어렵기 때문에 매우 위험한 신호이다.

- 은행은 고객이 원한다고 무조건 대출해줄 수 없다.

- 외부 법규를 반영하여 각 은행이 내부 여신규정과 지침을 만든다.
- 동일인은 개인 또는 법인 하나를 의미한다.
- 동일차주는 실질적으로 신용위험을 함께 부담하는 관계자들을 하나로 묶은 개념이다.
- 특정 고객이나 기업집단에 대출이 몰리지 않도록 신용공여 한도를 관리한다.
- 대주주와 임직원에 대한 특혜대출을 막기 위한 제한이 존재한다.

- 공공정보는 세금체납, 회생, 파산 등 공공기관이나 법적 절차와 관련된 정보이다.
- 연체대출금 보유자는 원칙적으로 신규여신이 제한된다.
- 기한의 이익을 상실하면 만기 전이라도 대출금 전액을 즉시 갚아야 할 수 있다.
- 분식회계나 부정적인 감사의견이 있는 기업은 실제 재무상태를 믿기 어려우므로 여신취급이 제한될 수 있다.

## 3. 금융상품 유형별 영업행위 준수

은행은 대출상품을 판매할 때 고객의 상황에 맞는 상품인지 확인하고, 중요한 내용을 충분히 설명해야 한다.

은행이 우월한 위치를 이용하여 고객에게 불리한 조건을 강요하거나, 대출을 조건으로 다른 금융상품에 가입하게 해서도 안 된다.

### 3-1. 적합성의 원칙

적합성의 원칙은 은행이 고객에게 대출상품 가입을 권유할 때 적용된다.

은행은 고객에게 대출을 권유하기 전에 다음 사항을 확인한다.

- 재산상황
- 소득
- 신용상태
- 기존 부채
- 변제계획
- 연령
- 대출 목적

확인 결과 고객이 대출금을 갚을 능력이 부족하다고 판단되면 해당 대출상품을 권유해서는 안 된다.

### 예시

월 소득이 250만 원인데 기존 대출 원리금으로 매월 180만 원을 내고 있는 고객에게 추가로 큰 금액의 대출을 권유하는 것은 적합하지 않을 수 있다.

### 핵심

> 은행이 먼저 대출을 권유하는 경우, 고객에게 맞지 않으면 권유하면 안 된다.

---

### 3-2. 적정성의 원칙

적정성의 원칙은 은행이 먼저 권유한 것이 아니라 고객이 직접 대출을 신청한 경우에 적용된다.

고객이 스스로 신청했더라도 은행이 확인한 결과 상환능력이 부족하거나 대출취급이 부적정하다고 판단되면 그 사실을 고객에게 알려야 한다.

은행은 고객에게 부적정하다는 사실을 알리고 고객의 확인을 받아야 한다.

### 예시

고객이 인터넷으로 직접 대출을 신청했지만 기존 부채가 많아 상환부담이 지나치게 높은 경우가 있다.

이 경우 은행은 해당 대출이 고객에게 적정하지 않을 수 있다는 사실을 알리고 확인을 받아야 한다.

### 적합성과 적정성의 차이

| 구분 | 적합성 | 적정성 |
|---|---|---|
| 신청 상황 | 은행이 상품을 권유 | 고객이 직접 신청 |
| 판단 결과 부적합한 경우 | 권유하면 안 됨 | 부적정 사실을 알리고 확인받음 |
| 핵심 | 권유 전 확인 | 직접 신청 후 경고 및 확인 |

### 쉽게 정리

- 적합성: 은행이 권유할 때 적용
- 적정성: 고객이 직접 신청할 때 적용

---

### 3-3. 설명의무

은행은 여신계약을 체결하기 전에 고객에게 대출상품의 중요한 내용을 설명해야 한다.

단순히 서류를 보여주는 것으로 끝나는 것이 아니라, 고객이 내용을 충분히 이해했다는 사실까지 확인해야 한다.

### 설명해야 하는 주요 내용

- 대출금리
- 고정금리인지 변동금리인지
- 중도상환수수료 부과 여부
- 중도상환수수료 부과기간
- 상환방법
- 상환금액
- 이자 납부시기
- 담보권 설정 여부
- 담보권 실행사유
- 담보권 실행 시 담보물의 소유권을 잃을 수 있다는 점
- 대출원금, 이자, 수수료 등 고객이 부담하는 총금액
- 대출이 신용에 미치는 영향
- 연체 시 적용되는 연체이자율
- 연체 시 발생할 수 있는 불이익
- 대출계약 철회
- 위법계약 해지
- 민원처리 및 분쟁조정 절차

### 예시

변동금리 대출이라면 현재 금리만 설명해서는 안 된다.

금리의 기준이 무엇인지, 금리가 변할 수 있다는 점, 금리가 오르면 고객의 이자부담도 커질 수 있다는 점을 함께 설명해야 한다.

### 개발자 관점

설명의무와 관련하여 다음과 같은 기능이 필요할 수 있다.

- 상품설명서 출력
- 상품설명서 전자교부
- 고객 확인 체크
- 설명완료 여부
- 전자서명
- 설명일시 저장
- 설명직원 저장
- 설명서 버전 저장
- 재설명 여부 관리

설명 당시 사용한 상품설명서와 약관의 버전을 함께 저장해야 나중에 어떤 내용을 설명했는지 확인할 수 있다.

---

### 3-4. 불공정영업행위의 금지

은행은 대출을 해주는 우월한 위치를 이용하여 고객에게 부당한 조건을 강요해서는 안 된다.

### 금지되는 주요 행위

#### 다른 금융상품 가입 강요

대출을 받으려면 예금, 적금, 보험, 펀드 등에 가입해야 한다고 강요해서는 안 된다.

#### 부당한 담보나 보증 요구

대출위험에 비해 지나치게 많은 담보나 보증을 요구해서는 안 된다.

#### 특정 상환방식 강요

은행이나 제3자의 이익을 위해 특정 상환방식만 사용하도록 강요해서는 안 된다.

#### 부당한 중도상환수수료 부과

허용되는 범위를 벗어나 중도상환수수료를 부과해서는 안 된다.

#### 개인대출의 제3자 연대보증 강요

개인에게 대출하면서 관계없는 제3자의 연대보증을 요구해서는 안 된다.

다만 법인의 대표자나 공동대표 등 일부 관계자는 예외적으로 인정될 수 있다.

### 예시

고객에게 다음과 같이 말하는 것은 문제가 될 수 있다.

> 이 대출을 받으려면 적금 50만 원짜리도 같이 가입해야 합니다.

대출과 관련 없는 금융상품 가입을 사실상 강요하기 때문이다.

---

### 3-5. 부당권유행위의 금지

은행은 대출계약을 권유할 때 고객이 잘못 판단하도록 만들거나 불리한 사실을 숨겨서는 안 된다.

### 부당권유에 해당할 수 있는 사례

- 확실하지 않은 내용을 확정된 사실처럼 설명
- 중요한 불이익을 숨김
- 고객이 원하지 않는데 반복적으로 가입을 권유
- 사실과 다른 내용을 설명
- 고객의 판단을 흐리게 하는 과장된 표현 사용
- 손실이나 위험이 없다고 설명

### 예시

변동금리 대출을 판매하면서 다음과 같이 말하면 문제가 될 수 있다.

> 금리는 앞으로도 거의 안 오르니까 걱정하지 않으셔도 됩니다.

금리변동 가능성을 확정적으로 판단하여 설명했기 때문이다.

---

### 3-6. 계약서류 제공의무

은행은 여신계약을 체결할 때 고객에게 계약 관련 서류를 제공해야 한다.

제공한 뒤에는 고객이 서류를 받았다는 확인도 받아야 한다.

### 주요 제공서류

- 은행여신거래기본약관
- 상품별 개별약관
- 여신거래약정서
- 기한연장 추가약정서
- 조건변경 추가약정서
- 채무인수약정서
- 담보관계 약정서
- 여신상품설명서

### 제공이 중요한 이유

고객은 계약 이후에도 다음 내용을 확인할 수 있어야 한다.

- 금리
- 만기
- 상환방법
- 담보내용
- 연체 시 불이익
- 기한의 이익 상실사유
- 수수료
- 고객과 은행의 권리와 의무

### 개발자 관점

- 서류 생성
- 서류 교부 여부
- 전자교부 동의
- 서류 수령확인
- 전자서명
- 문서 버전
- 교부일시
- 교부채널
- 재발급 이력

등을 관리할 수 있다.

---

### 3-7. 예금 등 구속행위의 금지

은행은 대출을 해주는 조건으로 고객의 예금이나 자금사용을 제한해서는 안 된다.

이러한 행위를 흔히 **꺾기**라고 한다.

### 주요 대상

- 중소기업
- 신용등급이 낮은 개인
- 중소기업 대표자
- 중소기업 등기임원

### 금지되는 주요 행위

#### 대출 전후 금융상품 가입 강요

대출실행 전후 일정 기간 안에 대출금액과 비교하여 지나치게 많은 예금, 적금, 보험, 펀드 등을 가입하게 해서는 안 된다.

#### 제3자 명의를 이용한 우회가입

고객 본인이 아닌 가족이나 관계회사 명의로 금융상품에 가입하게 하여 사실상 자금사용을 제한해서도 안 된다.

#### 다른 영업점이나 금융회사를 이용한 우회행위

대출을 실행한 영업점이 아닌 다른 영업점이나 다른 금융회사를 통해 금융상품 가입을 강요해서도 안 된다.

#### 예금인출 제한

고객의 동의 없이 예금에 담보권을 설정하거나 정당한 사유 없이 사고계좌로 등록하여 인출을 막아서는 안 된다.

### 예외적으로 가능한 경우

다음과 같이 대출과 직접 관련이 없거나 고객의 정상적인 필요에 따른 거래는 허용될 수 있다.

- 입출금이 자유로운 계좌 개설
- 법령상 반드시 필요한 상품
- 기업의 정상적인 대금결제 목적
- 담보로 제공한 예금의 범위 안에서 대출하는 경우
- 기존 예금 해지금액 범위 안에서 재예치
- 소액상품 가입
- 기업 내부수요를 위한 상품권 구입

### 핵심

> 고객이 대출금을 자유롭게 사용할 수 없도록 다른 상품에 묶어두는 행위는 금지된다.

---

### 개발자 관점

금융상품 영업행위 준수는 다음과 같은 화면과 기능에 반영될 수 있다.

### 상담 및 신청

- 권유 여부
- 고객 직접신청 여부
- 소득
- 재산상황
- 기존 부채
- 대출목적
- 변제계획
- 적합성 판단결과
- 적정성 판단결과

### 설명의무

- 설명항목 체크
- 설명완료 여부
- 고객 이해확인
- 전자서명
- 설명서 교부
- 설명서 버전
- 설명일시

### 불공정영업행위 방지

- 연대보증 요구 가능 여부
- 담보요구 적정성
- 중도상환수수료 적용기간
- 다른 상품 가입 여부
- 대출 전후 상품가입 이력

### 꺾기 점검

- 대출실행일
- 금융상품 가입일
- 월 납입금액
- 일시납입금액
- 상품유형
- 고객유형
- 예외사유
- 점검결과

---

### 핵심 정리

- 적합성은 은행이 상품을 권유할 때 적용된다.
- 적정성은 고객이 직접 신청했을 때 적용된다.
- 은행은 대출의 금리, 상환방법, 담보, 수수료, 연체불이익 등을 설명해야 한다.
- 고객이 설명을 이해했다는 확인까지 받아야 한다.
- 대출을 조건으로 다른 상품 가입을 강요해서는 안 된다.
- 고객에게 부당한 담보, 보증, 상환방식을 강요해서는 안 된다.
- 계약서류를 고객에게 제공하고 수령확인을 받아야 한다.
- 대출과 함께 예금이나 보험가입을 강요하는 행위를 꺾기라고 한다.

## 4. 여신거래기본약관

### 4-1. 약관이란

약관은 은행이 여러 고객과 반복적으로 계약하기 위해 미리 정해둔 공통 계약조건이다.

대출을 받을 때마다 은행과 고객이 모든 조건을 처음부터 하나씩 협의하기는 어렵다.

따라서 공통으로 적용할 내용을 미리 정해두고, 개별 대출의 금액, 금리, 기간 등은 별도의 약정서에 작성한다.

### 여신거래에서 사용되는 주요 문서

- 은행여신거래기본약관
- 상품별 약관
- 여신거래약정서
- 담보설정계약서
- 보증계약서
- 상품설명서

### 약관과 약정서의 차이

| 구분 | 약관 | 약정서 |
|---|---|---|
| 내용 | 여러 고객에게 공통 적용되는 조건 | 해당 고객의 개별 대출조건 |
| 예시 | 기한의 이익 상실, 상계, 통지 | 대출금액, 금리, 만기, 상환방법 |
| 적용 | 동일 유형 거래에 공통 적용 | 특정 계약에만 적용 |

### 쉽게 정리

> 약관은 공통규칙이고, 약정서는 고객별 계약내용이다.

---

### 4-2. 약관을 규제하는 이유

약관은 은행이 미리 작성하기 때문에 고객이 계약내용을 직접 협의하기 어렵다.

은행이 일방적으로 불리한 조건을 넣지 못하도록 약관은 법의 규제를 받는다.

은행은 다음 사항을 지켜야 한다.

- 약관을 고객이 볼 수 있도록 게시
- 고객이 요청하면 약관을 제공
- 중요한 내용을 설명
- 고객에게 지나치게 불리한 조항 금지
- 예상하기 어려운 조항을 일방적으로 적용하지 않음

---

### 4-3. 은행여신거래기본약관의 적용범위

은행여신거래기본약관은 은행과 채무자 사이의 여신거래에 공통으로 적용된다.

### 적용되는 주요 거래

- 가계자금대출
- 기업대출
- 어음대출
- 어음할인
- 당좌대출
- 지급보증
- 유가증권대여
- 외국환 관련 여신
- 그 밖의 여신거래

채무자는 단순히 대출을 받은 사람만 의미하지 않는다.

다음과 같이 은행에 채무를 부담하는 사람을 포함한다.

- 차주
- 어음할인 신청인
- 지급보증 신청인
- 보증인
- 채무인수인

---

### 4-4. 어음채권과 여신채권

어음을 이용한 여신거래에서는 은행이 두 가지 근거로 돈을 청구할 수 있다.

- 어음에 적힌 권리에 따른 청구
- 여신계약에 따른 청구

예를 들어 은행이 고객의 어음을 할인해줬는데 어음이 부도난 경우 은행은 어음상 권리나 여신계약상 권리를 이용하여 고객에게 돈을 청구할 수 있다.

---

### 4-5. 이자, 할인료, 보증료와 지연배상금

은행은 대출이나 지급보증 등의 거래에서 다음과 같은 금액을 받을 수 있다.

- 대출이자
- 어음할인료
- 지급보증료
- 수수료
- 지연배상금

### 지연배상금

지연배상금은 고객이 원금이나 이자를 약속한 날짜에 지급하지 않았을 때 추가로 부담하는 금액이다.

일반적으로 다음과 같이 생각할 수 있다.

> 미납금액 × 지연배상금률 × 연체일수

실제 계산은 상품과 은행 기준에 따라 달라질 수 있다.

### 고정금리와 변동금리

#### 고정금리

대출기간 중 금리가 원칙적으로 변하지 않는 방식이다.

#### 변동금리

기준금리 등의 변화에 따라 대출금리가 달라질 수 있는 방식이다.

은행이 금리를 변경할 때는 합리적인 기준과 정해진 절차를 따라야 한다.

### 고객에게 불리한 변경

은행의 변경으로 고객에게 예상하지 못한 불이익이 생기면 고객이 일정 기간 안에 계약을 해지할 수 있는 경우도 있다.

---

### 4-6. 비용의 부담

고객의 채무불이행으로 인해 은행이 채권을 회수하거나 담보권을 지키기 위한 비용이 발생할 수 있다.

이러한 비용은 채무자가 부담할 수 있다.

### 주요 비용

- 채권보전을 위한 가압류·가처분 비용
- 담보권 행사 비용
- 담보물 조사비용
- 담보물 추심비용
- 채무이행 독촉을 위한 통지비용
- 은행이 대신 지급한 비용

### 예시

고객이 대출금을 갚지 않아 은행이 부동산에 가압류를 신청했다면 가압류 관련 비용을 고객에게 청구할 수 있다.

---

### 4-7. 담보의 제공과 보충

고객이나 보증인의 신용상태가 나빠지거나 담보가치가 크게 떨어진 경우 은행은 담보를 추가로 요구할 수 있다.

### 예시

은행이 5억 원짜리 부동산을 담보로 3억 원을 대출했다.

이후 부동산 가치가 2억 원까지 떨어진다면 기존 담보만으로 대출금 회수가 어려워질 수 있다.

이 경우 은행은 다음을 요구할 수 있다.

- 추가 담보 제공
- 보증인 추가
- 대출금 일부 상환
- 다른 담보로 교체

---

### 4-8. 기한의 이익과 기한의 이익 상실

### 기한의 이익

채무자가 대출 만기일까지 대출금 전액을 갚지 않아도 되는 이익이다.

예를 들어 만기가 2030년이라면 고객이 계약을 정상적으로 지키는 동안 은행은 그 전에 전액상환을 요구할 수 없다.

### 기한의 이익 상실

고객이 연체하거나 약정을 위반하면 만기 전이라도 대출금 전액을 바로 갚아야 할 수 있다.

### 대표적인 사유

- 이자 장기연체
- 분할상환금 연속 미납
- 어음이나 수표 부도
- 파산 또는 회생절차
- 압류 또는 경매
- 담보가치의 심각한 하락
- 허위자료 제출
- 신용정보상 중대한 문제 발생
- 담보제공 의무 위반
- 보증인 교체 요구 불이행
- 그 밖의 중요한 약정 위반

### 기한의 이익 상실 효과

- 대출금 전액 즉시상환 의무
- 연체대출금 분류
- 보증인에 대한 청구
- 예금과 대출금의 상계
- 담보권 실행
- 강제집행

---

### 4-9. 기한의 이익 상실 통지

기한의 이익을 상실한 경우 은행은 정해진 절차에 따라 채무자나 연대보증인에게 그 내용을 통지해야 한다.

통지에는 다음 내용이 포함될 수 있다.

- 기한의 이익 상실사유
- 상실일자
- 상환해야 할 금액
- 상환기한
- 연체 시 불이익
- 담보권 실행 가능성

### 개발자 관점

- 통지대상자
- 통지일자
- 통지방법
- 도달 여부
- 재통지 여부
- 통지문서
- 보증인 통지 여부

를 관리할 수 있다.

---

### 4-10. 할인어음의 환매

은행이 어음을 할인해줬는데 어음이 정상적으로 지급되지 않으면, 어음할인을 받은 고객이 은행에 돈을 다시 갚아야 할 수 있다.

이를 환매채무라고 한다.

### 예시

A회사가 B회사의 어음을 은행에 할인하여 9,800만 원을 받았다.

만기에 B회사가 어음금액을 지급하지 못하면 은행은 A회사에 어음금액의 상환을 요구할 수 있다.

---

### 4-11. 상계

상계는 은행과 고객이 서로 돈을 받을 권리와 갚을 의무를 가지고 있을 때 이를 같은 금액만큼 없애는 것이다.

### 예시

고객이 은행에 갚아야 할 연체대출금이 1,000만 원이고, 같은 은행 예금계좌에 600만 원이 있다고 가정한다.

은행은 일정한 요건과 절차에 따라 예금 600만 원과 대출채무 600만 원을 상계할 수 있다.

상계 후 남는 대출채무는 400만 원이다.

### 핵심

> 은행이 고객에게 돌려줄 예금과 고객이 은행에 갚아야 할 대출금을 서로 맞춰 없애는 것이다.

---

### 4-12. 신고와 통지의무

고객의 정보가 변경되면 은행에 알려야 한다.

### 주요 변경사항

- 주소
- 연락처
- 이름 또는 법인명
- 대표자
- 사업장
- 법인등록사항
- 재산상태
- 신용상태
- 담보 관련 사항

고객이 변경사항을 알리지 않아 은행의 통지를 받지 못한 경우 고객에게 불이익이 생길 수 있다.

---

### 4-13. 약관변경

은행이 약관을 변경하는 경우 고객이 변경내용을 확인할 수 있도록 게시하거나 통지해야 한다.

고객에게 중요한 영향을 주는 변경은 적용시기와 내용을 명확하게 안내해야 한다.

### 개발자 관점

- 약관 버전
- 시행일
- 종료일
- 변경내용
- 변경사유
- 고객 통지 여부
- 동의 필요 여부
- 기존계약 적용 여부

를 관리할 수 있다.

---

### 개발자 관점

### 주요 관리정보

- 약관코드
- 약관명
- 약관버전
- 적용시작일
- 적용종료일
- 상품코드

- 이자율
- 지연배상금률

- 기한이익 상실 여부
- 상계 여부
- 통지이력

### 주요 상태값 예시

- 정상
- 연체
- 독촉
- 기한이익상실 예정
- 기한이익상실
- 담보보충 요청
- 상계

- 종결

### 중요한 이력

- 약정 당시 약관 버전
- 금리 변경이력
- 고객 통지이력
- 담보보충 요구이력
- 기한이익 상실이력

### 핵심 정리

- 약관은 여러 고객에게 공통 적용되는 계약조건이다.
- 약정서는 고객별 대출금액, 금리, 만기 등을 정한 개별계약이다.
- 은행은 약관을 고객이 확인할 수 있도록 해야 한다.
- 이자, 할인료, 보증료, 수수료, 지연배상금의 기준이 약관에 포함된다.
- 신용이 악화되거나 담보가치가 떨어지면 추가 담보를 요구할 수 있다.
- 연체나 약정 위반 시 기한의 이익을 상실할 수 있다.
- 기한의 이익을 상실하면 대출금 전액을 즉시 갚아야 할 수 있다.
- 은행은 일정한 요건에 따라 예금과 연체대출금을 상계할 수 있다.
- 약관, 금리, 통지, 기한이익 상실 등은 반드시 이력을 관리해야 한다.

## 5. 신용정보관리

### 5-1. 신용정보관리의 의미

은행은 대출이나 신용카드 거래를 하기 전에 고객의 신용상태를 확인해야 한다.

고객이 현재 어느 정도의 대출을 보유하고 있는지, 연체나 부도기록이 있는지, 보증채무가 있는지 등을 확인하여 대출 가능 여부와 조건을 결정한다.

한국신용정보원은 금융기관이 고객의 신용도를 판단하는 데 필요한 여러 신용정보를 모아 관리한다.

### 신용정보가 필요한 이유

- 고객의 상환능력 판단
- 기존 부채 확인
- 연체 및 부도 확인
- 동일 고객의 여러 금융기관 거래 확인
- 대출한도 산정
- 금리 산정
- 부실 가능성 판단
- 대출 실행 후 사후관리

---

### 5-2. 신용거래정보

신용거래정보는 고객이 어떤 금융거래를 하고 있는지를 보여주는 정보이다.

쉽게 말하면 다음을 확인할 수 있는 정보이다.

> 어디서, 언제, 어떤 금융상품을 이용했고 얼마의 채무를 가지고 있는가

### 개설·발급정보

- 당좌예금 개설 및 해지
- 가계당좌예금 개설 및 해지
- 신용카드 발급 및 해지
- 체크카드 발급 및 해지

### 개인대출정보

- 대출일자
- 대출금액
- 대출종류
- 대출잔액
- 할부금융 이용내역

### 개인채무보증정보

- 보증일자
- 보증금액
- 보증대상 채무
- 보증잔액

### 신용카드대출정보

- 카드론 이용일자
- 카드론 이용금액
- 현금서비스 이용일자
- 현금서비스 이용금액

### 기업신용공여정보

- 개인사업자 대출현황
- 법인 대출현황
- 한도성대출 현황
- 담보현황
- 법인 채무보증
- 채무인수약정
- 기업어음 거래
- 계정과목별 잔액
- 만기구조

### 복수카드정보

여러 카드사에서 카드를 발급받은 고객의 카드이용현황이다.

- 신용판매 이용금액
- 현금서비스 이용금액
- 리볼빙 잔액
- 카드이용한도
- 현금서비스한도

### 사고회원정보

- 카드 도난
- 카드 분실
- 카드 배달사고
- 보상신청
- 보상처리 결과

---

### 5-3. 신용도판단정보

신용도판단정보는 고객이 금융거래에서 돈을 제대로 갚았는지, 금융질서를 위반한 사실이 있는지를 판단할 수 있는 정보이다.

### 주요 정보

- 연체정보
- 대위변제정보
- 대지급정보
- 부도정보
- 금융질서문란정보
- 관련인정보
- 미수발생정보
- 무담보 미수채권정보

### 연체정보

대출원금, 이자, 카드대금 등을 약속한 날짜에 갚지 않은 정보이다.

### 대위변제정보

보증기관이 고객을 대신하여 금융기관에 돈을 갚은 정보이다.

### 대지급정보

은행이 지급보증 등에 따라 고객 대신 돈을 지급한 정보이다.

### 부도정보

어음이나 수표가 정상적으로 지급되지 않은 정보이다.

### 금융질서문란정보

- 대출사기
- 명의도용
- 허위서류 제출
- 카드부정사용
- 금융거래질서를 심각하게 해친 행위

### 관련인정보

부도기업의 대표자처럼 부실거래와 밀접한 관련이 있는 사람의 정보이다.

---

### 5-4. 공공정보

공공정보는 국가기관이나 공공기관이 관리하는 체납, 회생, 파산 등과 관련된 정보이다.

### 주요 공공정보

- 국세 체납
- 지방세 체납
- 관세 체납
- 과태료 체납
- 산재보험료 체납
- 고용보험료 체납
- 채무불이행자명부 등재
- 회생절차
- 개인회생절차
- 파산면책
- 신용회복지원 확정

### 신용도판단정보와 공공정보의 차이

| 구분 | 신용도판단정보 | 공공정보 |
|---|---|---|
| 발생영역 | 금융거래 | 공공기관·법원·행정기관 |
| 대표 사례 | 연체, 부도, 대위변제 | 세금체납, 회생, 파산 |
| 판단 목적 | 금융거래 이행 여부 | 공적 채무 및 법적 절차 확인 |

### 5-5. 신용능력정보

신용능력정보는 고객이 돈을 갚을 수 있는 능력을 판단하는 데 필요한 정보이다.

### 개인

- 소득
- 직업
- 근무기간
- 재산
- 납세실적
- 부채

### 기업

- 재무제표
- 매출액
- 자산
- 부채
- 영업이익
- 현금흐름
- 납세실적
- 신용등급

신용도판단정보가 과거에 돈을 잘 갚았는지를 보여준다면, 신용능력정보는 앞으로 돈을 갚을 능력이 있는지를 판단하는 데 사용된다.

### 5-6. 신용정보의 활용

은행은 신용정보를 단순히 조회만 하는 것이 아니라 여신업무 전반에서 사용한다.

### 상담 단계

- 기존대출 확인
- 연체 여부 확인
- 대략적인 대출 가능성 판단

### 신청 단계

- 신용정보조회 동의
- 고객 식별
- 외부신용정보 조회

### 심사 단계

- 신용등급 산정
- 상환능력 평가
- 한도 결정
- 금리 결정
- 승인 또는 거절

### 실행 단계

- 신규대출 등록
- 대출금액 등록
- 보증정보 등록

### 사후관리 단계

- 대출잔액 변경
- 연체 등록
- 정상화
- 대출상환
- 해지
- 채무보증 변경

### 5-7. 신용정보 동의

은행이 고객의 신용정보를 조회하거나 이용하려면 필요한 동의를 받아야 한다.

### 주요 동의

- 개인신용정보 수집·이용 동의
- 개인신용정보 조회 동의
- 개인신용정보 제공 동의
- 고유식별정보 처리 동의

동의서에는 다음 내용이 포함될 수 있다.

- 정보를 이용하는 목적
- 수집하는 정보의 항목
- 정보를 제공받는 기관
- 보유 및 이용기간
- 동의 거부 가능 여부
- 동의 거부 시 불이익

### 개발자 관점

- 동의서 버전
- 동의일시
- 동의채널
- 동의여부
- 전자서명
- 철회여부
- 동의 유효기간

을 관리할 수 있다.

### 5-8. 신용정보 관리 시 주의사항

신용정보는 고객의 중요한 개인정보이므로 업무상 필요한 범위에서만 사용해야 한다.

### 주의사항

- 업무와 관련 없는 조회 금지
- 타인의 신용정보 무단조회 금지
- 조회결과 외부유출 금지
- 필요한 기간보다 오래 보관하지 않음
- 잘못된 정보 발견 시 정정
- 사용자별 조회권한 관리
- 조회이력 보관
- 출력 및 다운로드 통제

### 예시

은행 직원이 친구의 대출내역이 궁금하다는 이유로 신용정보를 조회하면 안 된다.

업무상 담당하는 대출심사와 관련하여 필요한 경우에만 조회해야 한다.

## 1. 여신의 의미

### 1-1. 법률적인 의미

여신은 은행이 고객에게 돈을 직접 빌려주는 **대출보다 넓은 개념**이다.

「은행법」에서는 여신, 즉 신용공여를 은행이 금융거래 과정에서 직접 또는 간접적으로 신용위험을 부담하는 거래라고 본다.

여신에는 다음과 같은 거래가 포함된다.

- 대출
- 어음할인
- 지급보증
- 유가증권 매입
- 유가증권 대여
- 그 밖에 은행이 신용위험을 부담하는 거래

여신인지 판단할 때 중요한 것은 은행의 돈이 지금 바로 지급되는지 여부가 아니다.

> 고객이나 거래상대방이 약속을 지키지 못했을 때 은행에 손실이 발생할 가능성이 있는지가 중요하다.

따라서 다음과 같이 정리할 수 있다.

> 모든 대출은 여신에 포함되지만, 모든 여신이 대출인 것은 아니다.

### 1-2. 여신거래의 예시

### 대출

은행이 고객에게 돈을 직접 빌려주고, 고객은 정해진 기간 동안 원금과 이자를 갚는 거래이다.

예를 들어 은행이 고객에게 1억 원을 빌려줬는데 고객이 원금과 이자를 갚지 못하면 은행에 손실이 발생한다.

### 지급보증

고객이 다른 사람이나 회사에 부담하는 의무를 지키지 못할 경우, 은행이 대신 지급하겠다고 약속하는 거래이다.

지급보증을 해준 시점에는 은행의 돈이 바로 나가지 않는다.

하지만 고객이 의무를 지키지 못하면 은행이 대신 돈을 지급해야 하므로 여신에 포함된다.

### 어음할인

어음은 일정한 날짜에 돈을 지급하겠다는 내용이 적힌 증서이다.

고객이 몇 개월 뒤 받을 어음을 가지고 있지만 지금 당장 돈이 필요한 경우가 있다.

이때 은행이 어음을 넘겨받고, 만기까지의 할인료를 뺀 금액을 고객에게 먼저 지급하는 거래가 어음할인이다.

은행은 어음 만기일에 어음에 적힌 금액을 받는다.

어음이 부도나거나 지급되지 않으면 은행이 돈을 회수하지 못할 수 있으므로 어음할인도 여신에 포함된다.

### 1-3. 은행이 지급보증을 하는 이유

은행은 지급보증을 제공하고 고객으로부터 보증료를 받는다.

고객이 약속한 의무를 정상적으로 이행하면 은행은 실제 자금을 지급하지 않고 보증료 수익을 얻을 수 있다.

지급보증의 흐름은 다음과 같다.

1. 고객이 계약 이행이나 채무 지급을 위해 은행에 지급보증을 신청한다.
2. 은행이 고객의 신용상태와 상환능력을 심사한다.
3. 심사 결과 문제가 없으면 은행이 보증서를 발급한다.
4. 고객은 은행에 보증료를 납부한다.
5. 고객이 의무를 정상적으로 이행하면 은행의 실제 자금지출은 발생하지 않는다.
6. 고객이 의무를 이행하지 못하면 은행이 대신 지급한다.
7. 은행은 대신 지급한 금액을 고객에게 다시 청구한다.

은행이 대신 지급한 금액을 고객이 갚지 못하면 은행에 손실이 발생한다.

따라서 지급보증은 다음 두 가지 성격을 함께 가진다.

- 은행이 보증료를 받을 수 있는 수익거래
- 고객이 의무를 이행하지 못하면 손실이 발생할 수 있는 여신거래

이 때문에 은행은 지급보증도 대출처럼 신용심사와 한도관리를 진행한다.

### 1-4. 어음할인과 대출의 차이

대출과 어음할인은 모두 은행이 고객에게 먼저 돈을 지급한다는 점에서는 비슷하다.

하지만 돈을 지급하는 근거와 회수방법이 다르다.

| 구분 | 대출 | 어음할인 |
|---|---|---|
| 거래의 근거 | 은행과 고객이 대출계약을 체결 | 고객이 보유한 어음을 은행이 매입 |
| 고객이 받는 돈 | 대출금 | 어음금액에서 할인료를 뺀 금액 |
| 은행의 수익 | 대출이자 | 할인료 |
| 원래 돈을 지급할 사람 | 대출받은 고객 | 어음에 적힌 지급인 |
| 주요 위험 | 고객이 원금과 이자를 갚지 못함 | 어음이 부도나거나 지급되지 않음 |

### 예시

A회사가 B회사에 물건을 판매하고, 3개월 뒤 1억 원을 지급받기로 한 어음을 받았다고 가정한다.

하지만 A회사는 지금 당장 운영자금이 필요하다.

A회사는 어음을 은행에 넘기고, 은행은 할인료를 제외한 9,800만 원을 먼저 지급한다.

3개월 뒤에는 B회사가 은행에 어음금액 1억 원을 지급한다.

이 거래가 어음할인이다.

반면 대출은 고객이 가진 어음을 은행에 넘기는 것이 아니라, 고객과 은행이 직접 대출계약을 체결하고 고객이 은행에 원금과 이자를 갚는 거래이다.

### 1-5. 여신의 기능

은행은 고객의 예금 등을 통해 자금을 마련하고, 그 자금을 대출과 같은 여신으로 운용한다.

기본적인 흐름은 다음과 같다.

> 고객에게서 예금을 받음  
> → 필요한 고객이나 기업에 자금을 빌려줌  
> → 원금과 이자를 돌려받음  
> → 은행의 수익이 발생함

### 은행 측면

은행 입장에서 여신은 자금을 운용하고 수익을 얻는 주요 업무이다.

은행은 예금 등을 통해 조달한 자금을 대출 등으로 운용하고, 이자와 수수료를 통해 수익을 얻는다.

### 국가경제 측면

국가경제 측면에서 여신은 돈이 남는 사람의 자금을 돈이 필요한 개인이나 기업에 전달하는 역할을 한다.

예를 들면 다음과 같다.

- 개인이 은행에 예금한다.
- 기업이 은행에서 대출받아 공장이나 장비를 구입한다.
- 개인이 주택구입자금을 대출받는다.
- 사업자가 운영자금을 대출받아 사업을 계속한다.

은행은 이러한 과정을 통해 다음과 같은 기능을 수행한다.

- 자금의 매개
- 신용의 창조
- 자금의 배분
- 경제활동 지원

### 1-6. 여신을 신중하게 취급해야 하는 이유

은행이 빌려주는 돈은 대부분 은행의 자기 돈이 아니라 고객이 맡긴 예금에서 나온다.

은행이 상환능력이 없는 고객에게 무리하게 대출하고 돈을 돌려받지 못하면 다음과 같은 문제가 발생할 수 있다.

- 은행에 손실이 발생한다.
- 은행의 재무상태가 나빠진다.
- 예금자에게 영향을 줄 수 있다.
- 은행에 대한 신뢰가 떨어질 수 있다.
- 금융시장 전체에 문제가 생길 수 있다.

따라서 은행은 여신을 취급할 때 다음 사항을 확인해야 한다.

- 고객이 돈을 갚을 능력이 있는가
- 고객이 돈을 갚으려는 의지가 있는가
- 계약이 법적으로 문제가 없는가
- 고객의 위험에 맞는 금리를 적용했는가
- 원금과 이자를 제때 받을 수 있는 상환방식인가
- 담보나 보증이 필요한가

여신업무의 핵심은 단순히 대출을 많이 실행하는 것이 아니다.

> 돈을 빌려줘도 되는 고객을 잘 판단하고, 빌려준 돈을 제대로 회수하는 것이 핵심이다.

### 1-7. 여신운용의 기본원칙

여신운용의 4대 기본원칙은 다음과 같다.

- 안정성
- 수익성
- 성장성
- 공공성

### 안정성의 원칙

빌려준 돈을 안전하고 확실하게 회수할 수 있어야 한다는 원칙이다.

은행은 대출을 실행하기 전에 다음 사항을 확인한다.

- 고객의 소득이나 매출
- 기존에 보유한 대출
- 원금과 이자를 갚을 능력
- 돈을 갚으려는 의지
- 담보와 보증
- 특정 고객이나 업종에 대출이 몰려 있는지 여부

담보가 있다고 무조건 안전한 대출은 아니다.

담보는 고객이 대출금을 갚지 못했을 때 처분하여 돈을 회수하는 수단이다.

담보를 처분하는 데 시간이 오래 걸릴 수 있고, 담보가격이 떨어질 수도 있다.

따라서 은행은 담보보다 먼저 고객이 정상적인 소득이나 영업활동으로 대출금을 갚을 수 있는지를 확인해야 한다.

### 수익성의 원칙

은행이 부담하는 위험에 맞는 적절한 수익을 얻어야 한다는 원칙이다.

신용도가 좋은 고객과 연체 가능성이 높은 고객에게 같은 조건으로 대출해주면 은행이 부담하는 위험을 제대로 반영하지 못하게 된다.

위험이 높은 고객에게는 다음과 같은 조건이 적용될 수 있다.

- 대출금리가 높아질 수 있다.
- 승인금액이 줄어들 수 있다.
- 추가 담보나 보증을 요구할 수 있다.
- 대출이 거절될 수 있다.

은행은 고객의 신용위험, 대출기간, 담보 여부 등을 고려하여 금리와 한도를 결정한다.

### 성장성의 원칙

여신이 은행과 거래고객의 성장에 도움이 되어야 한다는 원칙이다.

기업은 대출받은 돈을 다음과 같이 사용할 수 있다.

- 공장이나 설비 구입
- 사업장 확대
- 원재료 구입
- 신규 사업 추진
- 운영자금 확보

기업이 성장하면 대출금을 갚을 능력도 좋아질 수 있고, 은행도 우량고객과의 거래를 확대할 수 있다.

다만 성장 가능성이 있다는 이유만으로 무조건 대출하는 것은 아니다.

성장 가능성과 실제 상환 가능성을 함께 검토해야 한다.

### 공공성의 원칙

은행의 여신은 은행의 수익뿐 아니라 국민경제에 미치는 영향도 고려해야 한다는 원칙이다.

은행이 특정 고객이나 업종에 돈을 지나치게 몰아주면 해당 고객이나 업종에 문제가 생겼을 때 은행에도 큰 손실이 발생할 수 있다.

또한 불법이나 투기를 위한 자금을 무분별하게 공급해서도 안 된다.

은행은 자금이 필요한 곳에 돈이 균형 있게 공급되고 경제발전에 도움이 되도록 여신을 운용해야 한다.

### 핵심 정리

- 여신은 대출보다 넓은 개념이다.
- 모든 대출은 여신이지만 모든 여신이 대출은 아니다.
- 은행이 손실을 볼 가능성이 있는 신용거래는 여신에 포함된다.
- 지급보증은 고객이 의무를 이행하지 못하면 은행이 대신 지급할 수 있으므로 여신이다.
- 어음할인은 미래에 받을 어음대금을 은행을 통해 미리 현금으로 바꾸는 거래이다.
- 은행은 예금으로 마련한 자금을 여신으로 운용하여 수익을 얻는다.
- 여신의 4대 기본원칙은 안정성, 수익성, 성장성, 공공성이다.
- 여신의 핵심은 상환 가능한 고객을 판단하고 빌려준 돈을 안전하게 회수하는 것이다.

## 2. 여신취급 금지 및 제한

---

### 여신취급을 제한하는 이유

은행은 고객이 원한다고 해서 무조건 돈을 빌려줄 수 없다.

은행이 빌려주는 돈의 대부분은 고객에게서 받은 예금이기 때문에, 위험한 고객이나 특정 기업에 대출이 지나치게 몰리면 은행의 건전성이 나빠질 수 있다.

따라서 다음과 같은 목적을 위해 법과 은행 내부규정으로 여신취급을 제한한다.

- 대출금이 특정 개인이나 기업에 지나치게 몰리는 것을 방지
- 은행의 대주주나 임직원이 특혜를 받는 것을 방지
- 상환 가능성이 매우 낮은 고객에게 추가 대출이 나가는 것을 방지
- 은행의 손실과 부실을 예방
- 예금자와 금융시장을 보호

여신취급 제한은 여신의 4대 기본원칙 중 특히 **안정성**과 **공공성**을 지키기 위한 제도이다.

### 2-1. 여신 관련 법규 및 규정

은행의 여신업무는 은행이 자체적으로 정한 기준만으로 처리하지 않는다.

은행은 외부 법규를 지켜야 하며, 각 은행은 해당 법규를 반영하여 내부 여신규정과 업무지침을 만든다.

여신업무와 관련된 대표적인 외부규정은 다음과 같다.

- 「은행법」
- 「은행법 시행령」
- 은행업감독규정
- 은행업감독업무시행세칙
- 그 밖의 금융 관련 법률 및 감독규정

이를 쉽게 구분하면 다음과 같다.

| 구분 | 의미 |
|---|---|
| 외규 | 법률, 시행령, 감독규정 등 은행 외부에서 정한 규정 |
| 내규 | 외규를 반영하여 각 은행이 내부적으로 만든 규정과 지침 |

외규에서 금지하는 거래는 은행 내부규정으로 허용할 수 없다.

은행 내부규정은 외부 법규를 기본으로 하면서, 은행의 위험관리정책에 따라 더 엄격하게 정할 수도 있다.

### 예시

법에서는 일정 한도까지 대출이 가능하더라도, 해당 은행의 내부정책상 위험이 높다고 판단되면 더 적은 금액만 승인하거나 대출을 거절할 수 있다.

[목차로 이동](#목차)

---

### 2-2. 은행법 등 관련 법규에 따른 제한

은행은 한 고객이나 특정 관계자에게 지나치게 많은 대출을 해서는 안 된다.

특정 고객에게 대출이 몰린 상태에서 그 고객이 부실해지면 은행에 큰 손실이 발생할 수 있기 때문이다.

이를 막기 위해 동일인, 동일차주, 대주주 등에 대한 신용공여 한도를 정하고 있다.

---

### 동일인과 동일차주에 대한 한도

#### 동일인

동일인은 한 명의 개인 또는 하나의 법인을 의미한다.

예를 들어 다음은 각각 하나의 동일인이다.

- 개인 고객 한 명
- A주식회사
- B재단법인

#### 동일차주

동일차주는 법적으로는 서로 다른 개인이나 법인이지만, 실질적으로 같은 신용위험을 공유하는 관계자들을 하나로 묶은 개념이다.

예를 들어 다음과 같은 경우이다.

- 같은 기업집단에 속한 회사들
- 서로 지급보증을 제공하는 관계회사
- 한 회사가 부실해지면 다른 회사도 함께 영향을 받는 관계
- 실질적으로 동일한 경영주가 지배하는 여러 회사

### 왜 따로 묶어서 보는가?

A회사와 B회사가 서로 다른 법인이라는 이유만으로 각각 큰 대출을 해주면, 실제로는 같은 기업집단에 대출이 몰릴 수 있다.

따라서 은행은 단순히 법인등록번호만 따로 보는 것이 아니라, 실질적으로 위험을 함께 부담하는 관계인지 확인한다.

### 한도

- 동일한 개인 또는 법인 각각에 대한 신용공여: 은행 자기자본의 20% 초과 금지
- 동일차주에 대한 신용공여: 은행 자기자본의 25% 초과 금지

### 예시

은행의 자기자본이 1조 원이라고 가정한다.

- 동일인 한도 20%: 최대 2,000억 원
- 동일차주 한도 25%: 최대 2,500억 원

A회사에 이미 1,500억 원을 대출했고, A회사와 동일차주인 B회사에 1,000억 원을 추가로 대출하려 한다면 동일차주 전체 금액은 2,500억 원이 된다.

이보다 더 대출하면 동일차주 한도를 초과할 수 있다.

> 실제 프로젝트에서는 현재 법령과 해당 은행의 내부규정을 기준으로 확인해야 한다.

---

### 거액 신용공여 한도

은행 자기자본의 10%를 초과하는 신용공여를 **거액 신용공여**라고 한다.

거액 신용공여의 총합계가 은행 자기자본의 5배를 넘지 못하도록 규제한다고 한다.
이는 개별 고객에 대한 한도뿐 아니라, 큰 대출 전체가 은행 자본에 비해 지나치게 많아지는 것도 막기 위한 규정이다.

### 예시


은행의 자기자본이 1조 원이라면 다음과 같다.

- 10%인 1,000억 원을 초과하는 여신은 거액 신용공여
- 거액 신용공여 전체 합계는 5조 원을 초과할 수 없음

즉 고객별 한도를 지켰더라도 큰 대출이 너무 많이 쌓이면 은행 전체가 위험해질 수 있으므로 총량도 관리한다.

---

### 대주주에 대한 신용공여 제한

대주주는 은행의 주요 주식을 보유하여 은행 경영에 영향력을 행사할 수 있는 사람이나 법인이다.

은행이 대주주에게 마음대로 큰 금액을 빌려주게 되면 다음과 같은 문제가 발생할 수 있다.

- 대주주가 자신의 지위를 이용하여 특혜대출을 받을 수 있음
- 정상적인 심사 없이 대출이 승인될 수 있음
- 대주주가 부실해질 경우 은행도 함께 위험해질 수 있음
- 예금자의 돈이 대주주의 사적인 이익을 위해 사용될 수 있음

따라서 대주주 개인뿐 아니라 전체 대주주에게 제공할 수 있는 신용공여에도 한도를 둔다.

핵심은 다음과 같다.

> 은행의 주인이 은행에 영향력을 행사하여 자기에게 유리한 대출을 받지 못하도록 제한하는 제도이다.

---

### 은행법상 금지업무

#### 해당 은행의 주식을 담보로 하는 대출

고객이 그 은행의 주식을 담보로 제공하고 같은 은행에서 돈을 빌리는 거래이다.

은행의 주가가 떨어지면 담보가치와 은행 건전성이 동시에 나빠질 수 있다.

#### 해당 은행의 주식을 사기 위한 대출

은행이 고객에게 돈을 빌려주고, 고객이 그 돈으로 해당 은행의 주식을 사게 하는 거래이다.

이는 은행이 대출을 이용하여 자기 주가나 자기자본을 인위적으로 부풀릴 위험이 있다.

#### 해당 은행 임직원에 대한 대출

은행 임직원은 자신의 업무상 지위를 이용하여 대출심사나 조건에 영향을 줄 수 있다.

따라서 특혜대출을 막기 위해 임직원 대출을 제한한다.

다만 금융위원회가 정한 소액대출 등 일부 예외는 존재할 수 있다.

### 2-3. 신용도판단정보 및 공공정보 등록 대상자

은행은 여신을 취급하기 전에 고객에게 심각한 연체나 공공기관 체납 등의 문제가 있는지 확인해야 한다.

이를 위해 한국신용정보원 등을 통해 신용도판단정보와 공공정보를 조회한다.

해당 정보가 있다고 해서 모든 경우에 법적으로 무조건 대출이 금지되는 것은 아니다.

은행은 정보의 종류, 등록사유, 해제 여부, 내부규정 등을 확인하여 다음과 같이 처리할 수 있다.

- 신규대출 제한
- 기한연장 제한
- 심사 강화
- 대출한도 축소
- 추가 담보나 보증 요구
- 대출 거절

---

### 신용도판단정보

신용도판단정보는 금융거래 과정에서 고객이 돈을 제대로 갚지 않았거나 금융질서를 위반한 사실과 관련된 정보이다.

대표적인 정보는 다음과 같다.

- 연체정보
- 대위변제·대지급정보
- 부도정보
- 금융질서문란정보
- 관련인정보
- 미수발생정보
- 신용거래의 무담보 미수채권정보

#### 연체정보

대출 원금이나 이자, 카드대금 등을 약속한 날짜에 갚지 않은 정보이다.

#### 대위변제·대지급정보

보증기관이나 은행 등이 고객을 대신하여 돈을 지급한 정보이다.

예를 들어 보증기관이 고객 대신 은행에 대출금을 갚았다면 대위변제가 발생한다.

#### 부도정보

어음이나 수표가 지급되지 않은 정보이다.

#### 금융질서문란정보

대출사기, 명의도용, 허위서류 제출 등 금융질서를 심각하게 해친 정보이다.

#### 관련인정보

부도기업의 대표자처럼 부실 거래와 밀접하게 관련된 사람에 대한 정보이다.

### 쉽게 정리하면

> 신용도판단정보는 금융거래에서 돈을 갚지 않았거나 금융질서를 위반한 기록이다.

---

### 공공정보

공공정보는 금융회사뿐 아니라 국가기관이나 공공기관이 보유한 체납, 회생, 파산 등의 정보이다.

대표적인 정보는 다음과 같다.

- 국세 체납
- 지방세 체납
- 과태료 체납
- 관세 체납
- 산재보험료·고용보험료 체납
- 채무불이행자명부 등재
- 회생절차 진행
- 개인회생절차 진행
- 파산면책 결정
- 신용회복지원 확정

### 신용도판단정보와 공공정보의 차이

| 구분 | 신용도판단정보 | 공공정보 |
|---|---|---|
| 주된 발생영역 | 금융거래 | 국가기관·공공기관의 행정 및 사법절차 |
| 대표적인 예 | 연체, 부도, 대위변제 | 세금체납, 회생, 파산, 신용회복 |
| 판단 의미 | 금융거래를 정상적으로 이행했는지 | 공적 채무나 법적 채무조정 상태가 있는지 |

### 예시

고객이 은행 대출이자를 오래 갚지 않았다면 **연체정보**에 해당한다.

고객이 국세를 내지 않았다면 **공공정보**에 해당한다.

고객이 개인회생절차를 진행하고 있다면 이 또한 **공공정보**에 해당한다.

### 2-4. 신용등급 불량기업

신용등급이 매우 낮은 기업은 대출금을 갚지 못할 가능성이 높다.

따라서 은행은 신용등급이 불량한 기업에 대해 다음과 같이 처리할 수 있다.

- 신규여신 취급 제한
- 기존 대출 회수
- 대출한도 축소
- 추가대출 중단
- 신용등급이 개선될 때까지 여신 감축
- 추가 담보나 보증 요구

여기서 중요한 점은 신용등급이 낮다고 무조건 기업이 망했다는 뜻은 아니라는 것이다.

하지만 은행 입장에서는 부실 가능성이 높아졌다는 신호이므로 더 보수적으로 심사한다.

### 예시

A기업의 매출이 크게 줄고 부채가 늘어나 신용등급이 하락했다.

이 경우 은행은 기존 한도대출을 그대로 연장하지 않고 다음과 같은 조건을 붙일 수 있다.

- 한도 일부 감액
- 일부 원금 상환
- 추가 담보 제공
- 만기 단축
- 신규대출 중단

### 2-5. 연체대출금 보유자 또는 은행에 손실을 끼친 자

대출을 이미 연체하고 있거나 과거에 은행에 실제 손실을 끼친 고객에게 다시 돈을 빌려주면 은행의 손실이 더 커질 수 있다.

따라서 연체를 정리하거나 은행의 손실을 회복하지 않은 고객은 원칙적으로 신규여신이 제한된다.

은행은 여신취급 전에 다음 사항을 확인한다.

- 현재 연체대출금을 보유하고 있는지
- 원금이나 이자를 미납했는지
- 기한의 이익을 상실한 대출이 있는지
- 과거에 원금을 감면받은 사실이 있는지
- 특수채권으로 관리된 이력이 있는지
- 은행에 실제 손실을 발생시킨 이력이 있는지

---

### 연체대출금의 범위

연체대출금은 단순히 대출 만기일이 지났는데 갚지 않은 경우만을 의미하지 않는다.

다음과 같은 경우도 포함된다.

- 상환기일이 지났지만 회수되지 않은 대출금
- 이자를 납부하지 않아 연체처리된 대출금
- 분할상환일에 갚지 않은 분할상환금
- 지급보증 후 은행이 대신 지급한 대지급금
- 그 밖에 기한의 이익을 상실한 대출금

### 예시

매월 100만 원씩 갚는 분할상환대출에서 이번 달 상환금 100만 원을 납부하지 않았다면, 해당 분할상환금은 연체에 해당한다.

대출 만기는 아직 2년 남았더라도 약정 위반으로 기한의 이익을 상실했다면 전체 대출금이 연체대출금으로 처리될 수 있다.

---

### 기한의 이익 상실

#### 기한의 이익

대출 만기일까지는 채무자가 대출금 전액을 갚지 않아도 되는 이익이다.

예를 들어 대출 만기가 2028년 12월 31일이라면, 채무자가 약정을 정상적으로 지키는 동안 은행은 그 전에 대출금 전액을 갚으라고 요구할 수 없다.

#### 기한의 이익 상실

연체나 약정 위반 등 정해진 사유가 발생하여 만기 전이라도 채무자가 대출금 전액을 즉시 갚아야 하는 상태이다.

#### 발생 가능한 효과

- 대출금이 연체대출금으로 분류됨
- 채무자에게 대출금 전액을 즉시 청구할 수 있음
- 보증인에게 변제를 청구할 수 있음
- 채무자의 예금과 대출금을 상계할 수 있음
- 담보권을 실행할 수 있음
- 강제집행 등 채권회수 절차를 진행할 수 있음

### 쉽게 정리하면

> 원래는 만기까지 기다려주기로 했지만, 고객이 약속을 어겼기 때문에 은행이 지금 전액을 갚으라고 요구할 수 있게 되는 것이다.

### 예외적인 여신취급

연체대출금을 보유한 고객은 원칙적으로 신규여신이 제한된다.

다만 새로운 여신을 취급해야 기존 대출을 정상화하거나 회수할 수 있는 특별한 상황이라면 은행 내부기준에 따라 예외취급이 가능할 수 있다.

예를 들어 기업이 일시적인 자금부족을 겪고 있지만 추가 운전자금을 지원하면 정상 영업과 기존 대출 상환이 가능하다고 판단되는 경우이다.

단, 이러한 예외취급은 일반적인 승인이 아니라 별도의 심사와 승인절차가 필요하다.

### 2-6. 부실회계기업

기업여신을 심사할 때 은행은 기업이 제출한 재무제표를 바탕으로 매출, 자산, 부채, 이익 등을 분석한다.

그런데 기업이 재무제표를 실제보다 좋게 보이도록 조작하면 은행은 잘못된 정보를 바탕으로 대출을 승인할 수 있다.

이러한 회계조작을 일반적으로 **분식회계**라고 한다.

### 예시

실제로는 손실이 발생했는데 매출이나 이익을 부풀려 흑자인 것처럼 표시하는 경우이다.

또는 실제 부채를 재무제표에서 누락하여 재무상태가 좋은 것처럼 보이게 할 수 있다.

이런 기업에 대출하면 은행은 실제 상환능력을 제대로 판단할 수 없으므로 위험하다.

### 감사의견

외부감사인은 기업의 재무제표를 확인한 뒤 감사의견을 제시한다.

대표적인 감사의견은 다음과 같다.

| 감사의견 | 의미 |
|---|---|
| 적정 | 재무제표가 중요한 부분에서 회계기준에 맞게 작성됨 |
| 한정 | 일부 문제는 있지만 재무제표 전체를 부정할 정도는 아님 |
| 부적정 | 재무제표의 문제가 매우 커서 적정하다고 볼 수 없음 |
| 의견거절 | 자료부족 등으로 감사인이 의견을 낼 수 없음 |

감사의견이 한정, 부적정 또는 의견거절이라면 은행은 해당 기업의 회계상 문제를 자세히 확인해야 한다.

특히 금융감독원으로부터 부실회계기업으로 지정된 기업은 여신취급이 제한될 수 있다.

### 왜 의견거절도 위험한가?

의견거절은 반드시 분식회계가 확인됐다는 뜻은 아니다.

하지만 감사인이 필요한 자료를 충분히 확인하지 못했거나 기업의 계속기업 가능성 등에 큰 불확실성이 있어 판단 자체를 할 수 없다는 의미일 수 있다.

은행 입장에서는 기업의 실제 재무상태를 믿기 어렵기 때문에 매우 위험한 신호이다.

- 은행은 고객이 원한다고 무조건 대출해줄 수 없다.

- 외부 법규를 반영하여 각 은행이 내부 여신규정과 지침을 만든다.
- 동일인은 개인 또는 법인 하나를 의미한다.
- 동일차주는 실질적으로 신용위험을 함께 부담하는 관계자들을 하나로 묶은 개념이다.
- 특정 고객이나 기업집단에 대출이 몰리지 않도록 신용공여 한도를 관리한다.
- 대주주와 임직원에 대한 특혜대출을 막기 위한 제한이 존재한다.

- 공공정보는 세금체납, 회생, 파산 등 공공기관이나 법적 절차와 관련된 정보이다.
- 연체대출금 보유자는 원칙적으로 신규여신이 제한된다.
- 기한의 이익을 상실하면 만기 전이라도 대출금 전액을 즉시 갚아야 할 수 있다.
- 분식회계나 부정적인 감사의견이 있는 기업은 실제 재무상태를 믿기 어려우므로 여신취급이 제한될 수 있다.

## 3. 금융상품 유형별 영업행위 준수

은행은 대출상품을 판매할 때 고객의 상황에 맞는 상품인지 확인하고, 중요한 내용을 충분히 설명해야 한다.

은행이 우월한 위치를 이용하여 고객에게 불리한 조건을 강요하거나, 대출을 조건으로 다른 금융상품에 가입하게 해서도 안 된다.

### 3-1. 적합성의 원칙

적합성의 원칙은 은행이 고객에게 대출상품 가입을 권유할 때 적용된다.

은행은 고객에게 대출을 권유하기 전에 다음 사항을 확인한다.

- 재산상황
- 소득
- 신용상태
- 기존 부채
- 변제계획
- 연령
- 대출 목적

확인 결과 고객이 대출금을 갚을 능력이 부족하다고 판단되면 해당 대출상품을 권유해서는 안 된다.

### 예시

월 소득이 250만 원인데 기존 대출 원리금으로 매월 180만 원을 내고 있는 고객에게 추가로 큰 금액의 대출을 권유하는 것은 적합하지 않을 수 있다.

### 핵심

> 은행이 먼저 대출을 권유하는 경우, 고객에게 맞지 않으면 권유하면 안 된다.

---

### 3-2. 적정성의 원칙

적정성의 원칙은 은행이 먼저 권유한 것이 아니라 고객이 직접 대출을 신청한 경우에 적용된다.

고객이 스스로 신청했더라도 은행이 확인한 결과 상환능력이 부족하거나 대출취급이 부적정하다고 판단되면 그 사실을 고객에게 알려야 한다.

은행은 고객에게 부적정하다는 사실을 알리고 고객의 확인을 받아야 한다.

### 예시

고객이 인터넷으로 직접 대출을 신청했지만 기존 부채가 많아 상환부담이 지나치게 높은 경우가 있다.

이 경우 은행은 해당 대출이 고객에게 적정하지 않을 수 있다는 사실을 알리고 확인을 받아야 한다.

### 적합성과 적정성의 차이

| 구분 | 적합성 | 적정성 |
|---|---|---|
| 신청 상황 | 은행이 상품을 권유 | 고객이 직접 신청 |
| 판단 결과 부적합한 경우 | 권유하면 안 됨 | 부적정 사실을 알리고 확인받음 |
| 핵심 | 권유 전 확인 | 직접 신청 후 경고 및 확인 |

### 쉽게 정리

- 적합성: 은행이 권유할 때 적용
- 적정성: 고객이 직접 신청할 때 적용

---

### 3-3. 설명의무

은행은 여신계약을 체결하기 전에 고객에게 대출상품의 중요한 내용을 설명해야 한다.

단순히 서류를 보여주는 것으로 끝나는 것이 아니라, 고객이 내용을 충분히 이해했다는 사실까지 확인해야 한다.

### 설명해야 하는 주요 내용

- 대출금리
- 고정금리인지 변동금리인지
- 중도상환수수료 부과 여부
- 중도상환수수료 부과기간
- 상환방법
- 상환금액
- 이자 납부시기
- 담보권 설정 여부
- 담보권 실행사유
- 담보권 실행 시 담보물의 소유권을 잃을 수 있다는 점
- 대출원금, 이자, 수수료 등 고객이 부담하는 총금액
- 대출이 신용에 미치는 영향
- 연체 시 적용되는 연체이자율
- 연체 시 발생할 수 있는 불이익
- 대출계약 철회
- 위법계약 해지
- 민원처리 및 분쟁조정 절차

### 예시

변동금리 대출이라면 현재 금리만 설명해서는 안 된다.

금리의 기준이 무엇인지, 금리가 변할 수 있다는 점, 금리가 오르면 고객의 이자부담도 커질 수 있다는 점을 함께 설명해야 한다.

### 개발자 관점

설명의무와 관련하여 다음과 같은 기능이 필요할 수 있다.

- 상품설명서 출력
- 상품설명서 전자교부
- 고객 확인 체크
- 설명완료 여부
- 전자서명
- 설명일시 저장
- 설명직원 저장
- 설명서 버전 저장
- 재설명 여부 관리

설명 당시 사용한 상품설명서와 약관의 버전을 함께 저장해야 나중에 어떤 내용을 설명했는지 확인할 수 있다.

---

### 3-4. 불공정영업행위의 금지

은행은 대출을 해주는 우월한 위치를 이용하여 고객에게 부당한 조건을 강요해서는 안 된다.

### 금지되는 주요 행위

#### 다른 금융상품 가입 강요

대출을 받으려면 예금, 적금, 보험, 펀드 등에 가입해야 한다고 강요해서는 안 된다.

#### 부당한 담보나 보증 요구

대출위험에 비해 지나치게 많은 담보나 보증을 요구해서는 안 된다.

#### 특정 상환방식 강요

은행이나 제3자의 이익을 위해 특정 상환방식만 사용하도록 강요해서는 안 된다.

#### 부당한 중도상환수수료 부과

허용되는 범위를 벗어나 중도상환수수료를 부과해서는 안 된다.

#### 개인대출의 제3자 연대보증 강요

개인에게 대출하면서 관계없는 제3자의 연대보증을 요구해서는 안 된다.

다만 법인의 대표자나 공동대표 등 일부 관계자는 예외적으로 인정될 수 있다.

### 예시

고객에게 다음과 같이 말하는 것은 문제가 될 수 있다.

> 이 대출을 받으려면 적금 50만 원짜리도 같이 가입해야 합니다.

대출과 관련 없는 금융상품 가입을 사실상 강요하기 때문이다.

---

### 3-5. 부당권유행위의 금지

은행은 대출계약을 권유할 때 고객이 잘못 판단하도록 만들거나 불리한 사실을 숨겨서는 안 된다.

### 부당권유에 해당할 수 있는 사례

- 확실하지 않은 내용을 확정된 사실처럼 설명
- 중요한 불이익을 숨김
- 고객이 원하지 않는데 반복적으로 가입을 권유
- 사실과 다른 내용을 설명
- 고객의 판단을 흐리게 하는 과장된 표현 사용
- 손실이나 위험이 없다고 설명

### 예시

변동금리 대출을 판매하면서 다음과 같이 말하면 문제가 될 수 있다.

> 금리는 앞으로도 거의 안 오르니까 걱정하지 않으셔도 됩니다.

금리변동 가능성을 확정적으로 판단하여 설명했기 때문이다.

---

### 3-6. 계약서류 제공의무

은행은 여신계약을 체결할 때 고객에게 계약 관련 서류를 제공해야 한다.

제공한 뒤에는 고객이 서류를 받았다는 확인도 받아야 한다.

### 주요 제공서류

- 은행여신거래기본약관
- 상품별 개별약관
- 여신거래약정서
- 기한연장 추가약정서
- 조건변경 추가약정서
- 채무인수약정서
- 담보관계 약정서
- 여신상품설명서

### 제공이 중요한 이유

고객은 계약 이후에도 다음 내용을 확인할 수 있어야 한다.

- 금리
- 만기
- 상환방법
- 담보내용
- 연체 시 불이익
- 기한의 이익 상실사유
- 수수료
- 고객과 은행의 권리와 의무

### 개발자 관점

- 서류 생성
- 서류 교부 여부
- 전자교부 동의
- 서류 수령확인
- 전자서명
- 문서 버전
- 교부일시
- 교부채널
- 재발급 이력

등을 관리할 수 있다.

---

### 3-7. 예금 등 구속행위의 금지

은행은 대출을 해주는 조건으로 고객의 예금이나 자금사용을 제한해서는 안 된다.

이러한 행위를 흔히 **꺾기**라고 한다.

### 주요 대상

- 중소기업
- 신용등급이 낮은 개인
- 중소기업 대표자
- 중소기업 등기임원

### 금지되는 주요 행위

#### 대출 전후 금융상품 가입 강요

대출실행 전후 일정 기간 안에 대출금액과 비교하여 지나치게 많은 예금, 적금, 보험, 펀드 등을 가입하게 해서는 안 된다.

#### 제3자 명의를 이용한 우회가입

고객 본인이 아닌 가족이나 관계회사 명의로 금융상품에 가입하게 하여 사실상 자금사용을 제한해서도 안 된다.

#### 다른 영업점이나 금융회사를 이용한 우회행위

대출을 실행한 영업점이 아닌 다른 영업점이나 다른 금융회사를 통해 금융상품 가입을 강요해서도 안 된다.

#### 예금인출 제한

고객의 동의 없이 예금에 담보권을 설정하거나 정당한 사유 없이 사고계좌로 등록하여 인출을 막아서는 안 된다.

### 예외적으로 가능한 경우

다음과 같이 대출과 직접 관련이 없거나 고객의 정상적인 필요에 따른 거래는 허용될 수 있다.

- 입출금이 자유로운 계좌 개설
- 법령상 반드시 필요한 상품
- 기업의 정상적인 대금결제 목적
- 담보로 제공한 예금의 범위 안에서 대출하는 경우
- 기존 예금 해지금액 범위 안에서 재예치
- 소액상품 가입
- 기업 내부수요를 위한 상품권 구입

### 핵심

> 고객이 대출금을 자유롭게 사용할 수 없도록 다른 상품에 묶어두는 행위는 금지된다.

---

### 개발자 관점

금융상품 영업행위 준수는 다음과 같은 화면과 기능에 반영될 수 있다.

### 상담 및 신청

- 권유 여부
- 고객 직접신청 여부
- 소득
- 재산상황
- 기존 부채
- 대출목적
- 변제계획
- 적합성 판단결과
- 적정성 판단결과

### 설명의무

- 설명항목 체크
- 설명완료 여부
- 고객 이해확인
- 전자서명
- 설명서 교부
- 설명서 버전
- 설명일시

### 불공정영업행위 방지

- 연대보증 요구 가능 여부
- 담보요구 적정성
- 중도상환수수료 적용기간
- 다른 상품 가입 여부
- 대출 전후 상품가입 이력

### 꺾기 점검

- 대출실행일
- 금융상품 가입일
- 월 납입금액
- 일시납입금액
- 상품유형
- 고객유형
- 예외사유
- 점검결과

---

### 핵심 정리

- 적합성은 은행이 상품을 권유할 때 적용된다.
- 적정성은 고객이 직접 신청했을 때 적용된다.
- 은행은 대출의 금리, 상환방법, 담보, 수수료, 연체불이익 등을 설명해야 한다.
- 고객이 설명을 이해했다는 확인까지 받아야 한다.
- 대출을 조건으로 다른 상품 가입을 강요해서는 안 된다.
- 고객에게 부당한 담보, 보증, 상환방식을 강요해서는 안 된다.
- 계약서류를 고객에게 제공하고 수령확인을 받아야 한다.
- 대출과 함께 예금이나 보험가입을 강요하는 행위를 꺾기라고 한다.

## 4. 여신거래기본약관

### 4-1. 약관이란

약관은 은행이 여러 고객과 반복적으로 계약하기 위해 미리 정해둔 공통 계약조건이다.

대출을 받을 때마다 은행과 고객이 모든 조건을 처음부터 하나씩 협의하기는 어렵다.

따라서 공통으로 적용할 내용을 미리 정해두고, 개별 대출의 금액, 금리, 기간 등은 별도의 약정서에 작성한다.

### 여신거래에서 사용되는 주요 문서

- 은행여신거래기본약관
- 상품별 약관
- 여신거래약정서
- 담보설정계약서
- 보증계약서
- 상품설명서

### 약관과 약정서의 차이

| 구분 | 약관 | 약정서 |
|---|---|---|
| 내용 | 여러 고객에게 공통 적용되는 조건 | 해당 고객의 개별 대출조건 |
| 예시 | 기한의 이익 상실, 상계, 통지 | 대출금액, 금리, 만기, 상환방법 |
| 적용 | 동일 유형 거래에 공통 적용 | 특정 계약에만 적용 |

### 쉽게 정리

> 약관은 공통규칙이고, 약정서는 고객별 계약내용이다.

---

### 4-2. 약관을 규제하는 이유

약관은 은행이 미리 작성하기 때문에 고객이 계약내용을 직접 협의하기 어렵다.

은행이 일방적으로 불리한 조건을 넣지 못하도록 약관은 법의 규제를 받는다.

은행은 다음 사항을 지켜야 한다.

- 약관을 고객이 볼 수 있도록 게시
- 고객이 요청하면 약관을 제공
- 중요한 내용을 설명
- 고객에게 지나치게 불리한 조항 금지
- 예상하기 어려운 조항을 일방적으로 적용하지 않음

---

### 4-3. 은행여신거래기본약관의 적용범위

은행여신거래기본약관은 은행과 채무자 사이의 여신거래에 공통으로 적용된다.

### 적용되는 주요 거래

- 가계자금대출
- 기업대출
- 어음대출
- 어음할인
- 당좌대출
- 지급보증
- 유가증권대여
- 외국환 관련 여신
- 그 밖의 여신거래

채무자는 단순히 대출을 받은 사람만 의미하지 않는다.

다음과 같이 은행에 채무를 부담하는 사람을 포함한다.

- 차주
- 어음할인 신청인
- 지급보증 신청인
- 보증인
- 채무인수인

---

### 4-4. 어음채권과 여신채권

어음을 이용한 여신거래에서는 은행이 두 가지 근거로 돈을 청구할 수 있다.

- 어음에 적힌 권리에 따른 청구
- 여신계약에 따른 청구

예를 들어 은행이 고객의 어음을 할인해줬는데 어음이 부도난 경우 은행은 어음상 권리나 여신계약상 권리를 이용하여 고객에게 돈을 청구할 수 있다.

---

### 4-5. 이자, 할인료, 보증료와 지연배상금

은행은 대출이나 지급보증 등의 거래에서 다음과 같은 금액을 받을 수 있다.

- 대출이자
- 어음할인료
- 지급보증료
- 수수료
- 지연배상금

### 지연배상금

지연배상금은 고객이 원금이나 이자를 약속한 날짜에 지급하지 않았을 때 추가로 부담하는 금액이다.

일반적으로 다음과 같이 생각할 수 있다.

> 미납금액 × 지연배상금률 × 연체일수

실제 계산은 상품과 은행 기준에 따라 달라질 수 있다.

### 고정금리와 변동금리

#### 고정금리

대출기간 중 금리가 원칙적으로 변하지 않는 방식이다.

#### 변동금리

기준금리 등의 변화에 따라 대출금리가 달라질 수 있는 방식이다.

은행이 금리를 변경할 때는 합리적인 기준과 정해진 절차를 따라야 한다.

### 고객에게 불리한 변경

은행의 변경으로 고객에게 예상하지 못한 불이익이 생기면 고객이 일정 기간 안에 계약을 해지할 수 있는 경우도 있다.

---

### 4-6. 비용의 부담

고객의 채무불이행으로 인해 은행이 채권을 회수하거나 담보권을 지키기 위한 비용이 발생할 수 있다.

이러한 비용은 채무자가 부담할 수 있다.

### 주요 비용

- 채권보전을 위한 가압류·가처분 비용
- 담보권 행사 비용
- 담보물 조사비용
- 담보물 추심비용
- 채무이행 독촉을 위한 통지비용
- 은행이 대신 지급한 비용

### 예시

고객이 대출금을 갚지 않아 은행이 부동산에 가압류를 신청했다면 가압류 관련 비용을 고객에게 청구할 수 있다.

---

### 4-7. 담보의 제공과 보충

고객이나 보증인의 신용상태가 나빠지거나 담보가치가 크게 떨어진 경우 은행은 담보를 추가로 요구할 수 있다.

### 예시

은행이 5억 원짜리 부동산을 담보로 3억 원을 대출했다.

이후 부동산 가치가 2억 원까지 떨어진다면 기존 담보만으로 대출금 회수가 어려워질 수 있다.

이 경우 은행은 다음을 요구할 수 있다.

- 추가 담보 제공
- 보증인 추가
- 대출금 일부 상환
- 다른 담보로 교체

---

### 4-8. 기한의 이익과 기한의 이익 상실

### 기한의 이익

채무자가 대출 만기일까지 대출금 전액을 갚지 않아도 되는 이익이다.

예를 들어 만기가 2030년이라면 고객이 계약을 정상적으로 지키는 동안 은행은 그 전에 전액상환을 요구할 수 없다.

### 기한의 이익 상실

고객이 연체하거나 약정을 위반하면 만기 전이라도 대출금 전액을 바로 갚아야 할 수 있다.

### 대표적인 사유

- 이자 장기연체
- 분할상환금 연속 미납
- 어음이나 수표 부도
- 파산 또는 회생절차
- 압류 또는 경매
- 담보가치의 심각한 하락
- 허위자료 제출
- 신용정보상 중대한 문제 발생
- 담보제공 의무 위반
- 보증인 교체 요구 불이행
- 그 밖의 중요한 약정 위반

### 기한의 이익 상실 효과

- 대출금 전액 즉시상환 의무
- 연체대출금 분류
- 보증인에 대한 청구
- 예금과 대출금의 상계
- 담보권 실행
- 강제집행

---

### 4-9. 기한의 이익 상실 통지

기한의 이익을 상실한 경우 은행은 정해진 절차에 따라 채무자나 연대보증인에게 그 내용을 통지해야 한다.

통지에는 다음 내용이 포함될 수 있다.

- 기한의 이익 상실사유
- 상실일자
- 상환해야 할 금액
- 상환기한
- 연체 시 불이익
- 담보권 실행 가능성

### 개발자 관점

- 통지대상자
- 통지일자
- 통지방법
- 도달 여부
- 재통지 여부
- 통지문서
- 보증인 통지 여부

를 관리할 수 있다.

---

### 4-10. 할인어음의 환매

은행이 어음을 할인해줬는데 어음이 정상적으로 지급되지 않으면, 어음할인을 받은 고객이 은행에 돈을 다시 갚아야 할 수 있다.

이를 환매채무라고 한다.

### 예시

A회사가 B회사의 어음을 은행에 할인하여 9,800만 원을 받았다.

만기에 B회사가 어음금액을 지급하지 못하면 은행은 A회사에 어음금액의 상환을 요구할 수 있다.

---

### 4-11. 상계

상계는 은행과 고객이 서로 돈을 받을 권리와 갚을 의무를 가지고 있을 때 이를 같은 금액만큼 없애는 것이다.

### 예시

고객이 은행에 갚아야 할 연체대출금이 1,000만 원이고, 같은 은행 예금계좌에 600만 원이 있다고 가정한다.

은행은 일정한 요건과 절차에 따라 예금 600만 원과 대출채무 600만 원을 상계할 수 있다.

상계 후 남는 대출채무는 400만 원이다.

### 핵심

> 은행이 고객에게 돌려줄 예금과 고객이 은행에 갚아야 할 대출금을 서로 맞춰 없애는 것이다.

---

### 4-12. 신고와 통지의무

고객의 정보가 변경되면 은행에 알려야 한다.

### 주요 변경사항

- 주소
- 연락처
- 이름 또는 법인명
- 대표자
- 사업장
- 법인등록사항
- 재산상태
- 신용상태
- 담보 관련 사항

고객이 변경사항을 알리지 않아 은행의 통지를 받지 못한 경우 고객에게 불이익이 생길 수 있다.

---

### 4-13. 약관변경

은행이 약관을 변경하는 경우 고객이 변경내용을 확인할 수 있도록 게시하거나 통지해야 한다.

고객에게 중요한 영향을 주는 변경은 적용시기와 내용을 명확하게 안내해야 한다.

### 개발자 관점

- 약관 버전
- 시행일
- 종료일
- 변경내용
- 변경사유
- 고객 통지 여부
- 동의 필요 여부
- 기존계약 적용 여부

를 관리할 수 있다.

---

### 개발자 관점

### 주요 관리정보

- 약관코드
- 약관명
- 약관버전
- 적용시작일
- 적용종료일
- 상품코드

- 이자율
- 지연배상금률

- 기한이익 상실 여부
- 상계 여부
- 통지이력

### 주요 상태값 예시

- 정상
- 연체
- 독촉
- 기한이익상실 예정
- 기한이익상실
- 담보보충 요청
- 상계

- 종결

### 중요한 이력

- 약정 당시 약관 버전
- 금리 변경이력
- 고객 통지이력
- 담보보충 요구이력
- 기한이익 상실이력

### 핵심 정리

- 약관은 여러 고객에게 공통 적용되는 계약조건이다.
- 약정서는 고객별 대출금액, 금리, 만기 등을 정한 개별계약이다.
- 은행은 약관을 고객이 확인할 수 있도록 해야 한다.
- 이자, 할인료, 보증료, 수수료, 지연배상금의 기준이 약관에 포함된다.
- 신용이 악화되거나 담보가치가 떨어지면 추가 담보를 요구할 수 있다.
- 연체나 약정 위반 시 기한의 이익을 상실할 수 있다.
- 기한의 이익을 상실하면 대출금 전액을 즉시 갚아야 할 수 있다.
- 은행은 일정한 요건에 따라 예금과 연체대출금을 상계할 수 있다.
- 약관, 금리, 통지, 기한이익 상실 등은 반드시 이력을 관리해야 한다.

## 5. 신용정보관리

### 5-1. 신용정보관리의 의미

은행은 대출이나 신용카드 거래를 하기 전에 고객의 신용상태를 확인해야 한다.

고객이 현재 어느 정도의 대출을 보유하고 있는지, 연체나 부도기록이 있는지, 보증채무가 있는지 등을 확인하여 대출 가능 여부와 조건을 결정한다.

한국신용정보원은 금융기관이 고객의 신용도를 판단하는 데 필요한 여러 신용정보를 모아 관리한다.

### 신용정보가 필요한 이유

- 고객의 상환능력 판단
- 기존 부채 확인
- 연체 및 부도 확인
- 동일 고객의 여러 금융기관 거래 확인
- 대출한도 산정
- 금리 산정
- 부실 가능성 판단
- 대출 실행 후 사후관리
